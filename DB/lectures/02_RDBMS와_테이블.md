# 2강 — RDBMS란? 테이블, 행, 열 개념

> **목표:** DB에 데이터가 "어떤 모양"으로 저장되는지 이해한다.
> **선수 지식:** 1강 (DB와 DBMS의 차이)

---

## 1. DBMS에도 종류가 있다

지난 시간에 "DBMS는 DB를 관리하는 소프트웨어"라고 배웠다. 그런데 DBMS도 **데이터를 저장하는 방식**에 따라 여러 종류로 나뉜다.

| 종류 | 데이터 저장 방식 | 대표 제품 |
|---|---|---|
| **관계형 (RDBMS)** | 표(테이블) 형태 | PostgreSQL, MySQL, Oracle |
| 문서형 (Document) | JSON 같은 문서 | MongoDB |
| 키-값 (Key-Value) | C++의 `TMap`처럼 | Redis |
| 그래프 (Graph) | 노드와 연결선 | Neo4j |

우리가 배우는 **PostgreSQL은 RDBMS** — **R**elational **D**ata**b**ase **M**anagement **S**ystem.
번역하면 "관계형 데이터베이스 관리 시스템"이다.

---

## 2. "관계형"이 뭔데?

핵심 단어 두 개만 기억하자.

1. **데이터를 표(테이블) 모양으로 저장**한다.
2. **표와 표를 서로 연결(관계)** 시킬 수 있다.

엑셀 시트를 떠올리면 거의 똑같다. 시트 하나하나가 테이블이고, 시트들끼리 ID로 연결되는 구조다.

---

## 3. 테이블 — DB에서 가장 중요한 개념

테이블(Table)은 RDBMS에서 데이터를 담는 **기본 단위**다.

예를 들어 게임 유저 정보를 저장하는 `users` 테이블은 이렇게 생겼다.

| id | name      | level | gold  |
|----|-----------|-------|-------|
| 1  | 김철수    | 12    | 500   |
| 2  | 이영희    | 45    | 12000 |
| 3  | 박민수    | 7     | 100   |

이게 끝이다. 진짜다. **DB는 결국 이런 표들의 모음**이다.

### C++로 비유하면?

```cpp
struct FUser
{
    int32   Id;
    FString Name;
    int32   Level;
    int32   Gold;
};

TArray<FUser> Users;  // 이게 곧 'users 테이블'
```

| C++ 개념        | DB 개념              |
|-----------------|---------------------|
| `struct` 정의    | 테이블 **스키마(구조)** |
| `struct` 인스턴스 1개 | **행(row)** 1개       |
| `struct`의 멤버 변수 | **열(column)**        |
| `TArray<FUser>` | 테이블 전체             |

---

## 4. 행(Row)과 열(Column) — 용어 정리

위 표를 다시 보자.

| id | name      | level | gold  |
|----|-----------|-------|-------|
| 1  | 김철수    | 12    | 500   |    ← **행 (Row)**
| 2  | 이영희    | 45    | 12000 |    ← **행 (Row)**

↑ 이게 **열(Column)**

- **열(Column):** 세로줄. 데이터의 **종류**를 정의. (`name`, `level` 같은 것)
  - 다른 말로 **필드(field)**, **속성(attribute)** 이라고도 부른다.
  - 각 열은 **데이터 타입**이 정해져 있다. (`name`은 문자열, `level`은 정수…)
- **행(Row):** 가로줄. 실제 데이터 **한 건**.
  - 다른 말로 **레코드(record)**, **튜플(tuple)** 이라고도 부른다.

> 💡 책에 따라 용어가 섞여 나오는데 다 같은 뜻이다. 헷갈리지 말자.
> - 열 = 컬럼 = 필드 = 속성
> - 행 = 로우 = 레코드 = 튜플

---

## 5. 스키마(Schema) — 테이블의 "설계도"

테이블을 만들기 전에, 먼저 **어떤 열이 어떤 타입으로 있을 건지**를 정의해야 한다. 이 설계도를 **스키마**라고 한다.

C++로 치면 `struct` 선언과 같다.

```cpp
// C++의 struct 선언 == DB의 스키마 정의
struct FUser
{
    int32   Id;      // → id    INTEGER
    FString Name;    // → name  TEXT
    int32   Level;   // → level INTEGER
    int32   Gold;    // → gold  INTEGER
};
```

PostgreSQL에서 위 스키마를 만드는 SQL은 이렇게 생겼다.

```sql
CREATE TABLE users (
    id    INTEGER,
    name  TEXT,
    level INTEGER,
    gold  INTEGER
);
```

> 어디서 본 듯한 느낌이지? `CREATE TABLE`이 `struct`, 괄호 안이 멤버 변수 선언. 거의 똑같다.

자세한 문법은 5강(`CREATE TABLE`)에서 본격적으로 다룬다. 지금은 **"테이블엔 설계도가 있다"** 만 기억하자.

---

## 6. "관계"는 어떻게 만드는가? — 살짝 맛보기

RDBMS의 **R(Relational, 관계형)** 답게, 테이블들을 서로 연결할 수 있다.

예를 들어 길드 시스템이 있다고 해보자.

**guilds 테이블**
| id | guild_name |
|----|-----------|
| 1  | 흑사자단  |
| 2  | 백호회    |

**users 테이블**
| id | name   | guild_id |
|----|--------|----------|
| 1  | 김철수 | 1        | ← 흑사자단 소속
| 2  | 이영희 | 2        | ← 백호회 소속
| 3  | 박민수 | 1        | ← 흑사자단 소속

`users.guild_id`가 `guilds.id`를 가리키고 있다. **이게 "관계"다.**

C++로 비유하면 포인터/참조와 비슷하다.

```cpp
struct FGuild { int32 Id; FString Name; };
struct FUser  { int32 Id; FString Name; int32 GuildId; }; // GuildId가 FGuild를 가리킴
```

이렇게 ID로 연결하는 방식을 **외래 키(Foreign Key)** 라고 부른다. 자세한 건 16강에서 다룬다.

---

## 7. 테이블 설계의 황금 규칙

지금은 외워두기만 해도 좋다.

1. **한 행 = 한 건의 데이터** (한 유저, 한 아이템, 한 거래 등)
2. **한 열 = 한 종류의 정보** (이름은 이름끼리, 레벨은 레벨끼리)
3. **모든 행은 서로 구분되어야 한다** — 보통 `id` 같은 **고유 번호** 부여
4. **하나의 칸엔 하나의 값** — 한 칸에 `"검, 방패, 포션"` 이렇게 몰아넣지 않음

마지막 규칙이 처음엔 어색할 수 있다. 보유 아이템 같은 "여러 개"는 어떻게 저장하나?
👉 **별도 테이블을 만들어서 관계로 연결한다.** (이게 4번째 규칙 = 정규화의 시작)

---

## 8. 이번 강의에서 기억할 것

1. PostgreSQL은 **RDBMS**, 데이터를 **표(테이블)** 형태로 저장한다.
2. 테이블은 **행(Row, 데이터 한 건)** 과 **열(Column, 데이터 종류)** 로 구성된다.
3. 테이블의 구조를 정의한 설계도가 **스키마**다.
4. 테이블끼리는 **ID로 연결**(관계)할 수 있다.
5. **한 행 = 한 건, 한 열 = 한 종류, 한 칸 = 한 값**.
