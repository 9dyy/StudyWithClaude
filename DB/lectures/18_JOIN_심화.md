# 18강 — `JOIN` 심화 (`RIGHT`/`FULL`/`CROSS`/`SELF`, 다중 JOIN)

> **목표:** JOIN의 나머지 종류, 자기 자신과의 JOIN, 그리고 3개 이상 테이블 묶기.
> **선수 지식:** 17강

---

## 1. JOIN 종류 한눈에

| JOIN | 결과 | 자주 쓰나 |
|------|------|----------|
| `INNER JOIN` | 양쪽 모두 매칭 | ★★★★★ |
| `LEFT JOIN` (= `LEFT OUTER JOIN`) | 왼쪽 전부 + 오른쪽 매칭 | ★★★★★ |
| `RIGHT JOIN` (= `RIGHT OUTER JOIN`) | 오른쪽 전부 + 왼쪽 매칭 | ★ (거의 LEFT로 바꿔 씀) |
| `FULL OUTER JOIN` | 양쪽 전부, 매칭 없으면 NULL | ★★ |
| `CROSS JOIN` | 모든 조합 (카티시안 곱) | ★ (특수 용도) |
| `SELF JOIN` | 자기 자신과 JOIN | ★★ (계층/비교) |

> 💡 `LEFT OUTER JOIN`처럼 `OUTER`를 명시해도 되지만, 실무에선 그냥 `LEFT JOIN`/`RIGHT JOIN`/`FULL JOIN`으로 짧게 쓴다.

---

## 2. `RIGHT JOIN`

`LEFT JOIN`의 좌우만 뒤집은 것. **오른쪽 테이블을 전부 살리고** 왼쪽에서 매칭되는 것만.

```sql
SELECT u.name, g.guild_name
FROM users u
RIGHT JOIN guilds g  ON u.guild_id = g.id;
```

결과:

```text
   name   | guild_name
----------+------------
 김철수   | 흑사자단
 이영희   | 흑사자단
 최백호   | 백호단
  NULL    | 청룡회       ← 가입자 없는 길드도 살아남음
```

`RIGHT JOIN`은 사실 거의 안 쓴다. **테이블 순서만 바꾸면 `LEFT JOIN`으로 표현 가능**하기 때문에:

```sql
-- 위와 완전 동일
SELECT u.name, g.guild_name
FROM guilds g
LEFT JOIN users u  ON u.guild_id = g.id;
```

쿼리를 읽을 때 **위에서 아래로 흐름이 일관**되도록 `LEFT JOIN`만 쓰는 게 관례.

---

## 3. `FULL OUTER JOIN`

**양쪽 모두 전부 살리고** 매칭 안 되는 쪽은 NULL.

```sql
SELECT u.name, g.guild_name
FROM users u
FULL OUTER JOIN guilds g  ON u.guild_id = g.id;
```

결과:

```text
   name    | guild_name
-----------+------------
 김철수    | 흑사자단
 이영희    | 흑사자단
 최백호    | 백호단
 박무소속  |   NULL        ← 좌측만 있는 행
  NULL     | 청룡회        ← 우측만 있는 행
```

언제 쓰나? **양쪽 데이터 차이를 검사**할 때.

```sql
-- 결제 안 된 주문 + 주문 없는 결제 (데이터 정합성 검증)
SELECT o.id AS order_id, p.id AS payment_id
FROM orders o
FULL OUTER JOIN payments p  ON o.id = p.order_id
WHERE o.id IS NULL OR p.id IS NULL;
```

> 💡 일상적인 조회보단 **데이터 검증/리포팅**에서 가끔 보는 정도.

---

## 4. `CROSS JOIN` — 모든 조합

**조건 없이 모든 행을 곱한다.** 카티시안 곱(Cartesian product).

```sql
SELECT u.name, g.guild_name
FROM users u
CROSS JOIN guilds g;
```

`users` 4행 × `guilds` 3행 = **12행** (모든 조합).

쓸 일이 거의 없지만, **모든 조합을 생성해야 할 때** 유용.

```sql
-- 모든 유저 × 모든 요일 = 1주일 출석 체크 슬롯 생성
SELECT u.id AS user_id, d.day_name
FROM users u
CROSS JOIN (VALUES ('월'),('화'),('수'),('목'),('금'),('토'),('일')) AS d(day_name);
```

> ⚠️ **`FROM users, guilds` 처럼 쉼표로 나열해도 `CROSS JOIN`이 된다.** 옛날 SQL 문법인데, 실수로 `JOIN` 빼먹으면 어마어마한 행이 나온다. **명시적으로 `CROSS JOIN`을 쓰자.**

---

## 5. `SELF JOIN` — 자기 자신과 JOIN

**같은 테이블을 두 번 JOIN.** 같은 테이블의 두 행을 비교하거나, 16강에서 본 **계층 구조**를 펼칠 때.

### 예시 1 — 계층 구조 (부모-자식)

16강의 `categories` 테이블 (자기 참조 FK):

```sql
CREATE TABLE categories (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name TEXT NOT NULL,
    parent_id BIGINT REFERENCES categories(id)
);

INSERT INTO categories (id, name, parent_id) VALUES
  (1, '게임', NULL),
  (2, 'RPG', 1),
  (3, 'FPS', 1),
  (4, '음악', NULL),
  (5, '클래식', 4);
```

자식 — 부모 이름을 같이 보고 싶다:

```sql
SELECT c.name AS child_name, p.name AS parent_name
FROM   categories c
LEFT JOIN categories p  ON c.parent_id = p.id;
```

결과:

```text
 child_name | parent_name
------------+-------------
 게임       |   NULL
 RPG        | 게임
 FPS        | 게임
 음악       |   NULL
 클래식     | 음악
```

**같은 테이블을 두 번 등장시키므로 별칭이 필수다.** `c` (자식 역할), `p` (부모 역할)로 구분.

> 💡 **2단계 깊이까지만** SELF JOIN으로 처리 가능. 무제한 깊이는 28강의 `WITH RECURSIVE`.

### 예시 2 — 같은 테이블의 두 행 비교

```sql
-- 같은 길드 안에서 나보다 레벨 높은 동료가 있는 유저
SELECT me.name AS my_name, peer.name AS senior_name
FROM users me
JOIN users peer
  ON me.guild_id = peer.guild_id   -- 같은 길드
 AND peer.level  > me.level;       -- 레벨 더 높음
```

별칭 `me`, `peer`로 같은 테이블의 두 역할을 구분.

## 6. `USING` 절 — JOIN 조건의 단축형

JOIN하는 **두 컬럼의 이름이 같으면** `USING (컬럼)`으로 짧게 쓸 수 있다.

```sql
-- ON 버전
SELECT *
FROM order_items oi
JOIN orders o  ON oi.order_id = o.id;

-- USING 버전 (양쪽 다 컬럼 이름이 order_id일 때만)
SELECT *
FROM order_items oi
JOIN orders o  USING (order_id);
```

⚠️ 단, `USING`은 **이름이 정확히 같아야** 동작한다. `users.guild_id`와 `guilds.id`처럼 다르면 못 씀.

또 `USING`이 묶은 컬럼은 결과에서 **한 번만 나온다** (`SELECT *`로 봤을 때).

```sql
SELECT * FROM a JOIN b USING (x);   -- x는 결과에 한 번만
SELECT * FROM a JOIN b ON a.x = b.x; -- x가 두 번 (a.x, b.x)
```

> 💡 `USING`이 짧지만 **`ON`이 더 명시적**이고 실수가 적다. 둘 다 알아두되, 기본은 `ON`.

### `NATURAL JOIN`은 ❌

`NATURAL JOIN`은 "같은 이름 컬럼 전부를 자동으로 JOIN 조건으로 삼는다." 편해 보이지만, 나중에 컬럼 추가만 해도 JOIN 의미가 조용히 바뀐다. **실무에선 쓰지 말 것.**

## 7. 다중 JOIN — 3개 이상의 테이블

JOIN은 **이어 붙일 수 있다.** 게임 DB의 세 테이블을 한 번에:

```sql
-- 유저 + 그 유저의 길드 + 그 유저의 아이템
SELECT
  u.name        AS user_name,
  g.guild_name  AS guild_name,
  i.item_name   AS item_name,
  i.quantity
FROM   users u
LEFT  JOIN guilds g  ON g.id      = u.guild_id
LEFT  JOIN items  i  ON i.user_id = u.id
ORDER BY u.id, i.id;
```

읽는 법: "**`users`에서 시작 → 각 유저에 길드를 LEFT JOIN → 각 유저에 아이템을 LEFT JOIN.**"

결과 (예시):

```text
 user_name | guild_name | item_name | quantity
-----------+------------+-----------+----------
 김철수    | 흑사자단   | 회복포션  |    5
 김철수    | 흑사자단   | 강철검    |    1
 이영희    | 흑사자단   | 마법서    |    3
 박무소속  |   NULL     | 나무막대  |    1
 최백호    | 백호단     |   NULL    |   NULL
```

### ⚠️ 행 개수가 폭증할 수 있다

유저가 아이템을 N개 가지고 있으면, **유저 행이 N번 반복**된다. 이건 JOIN의 본성. 집계로 묶거나(`GROUP BY` — 13강) 1행으로 줄이는 별도 처리가 필요.

```sql
-- 유저별 아이템 개수만 보고 싶다
SELECT u.name, COUNT(i.id) AS item_count
FROM   users u
LEFT JOIN items i  ON i.user_id = u.id
GROUP BY u.id, u.name
ORDER BY u.id;
```

> 💡 **`COUNT(i.id)`는 NULL을 세지 않는다** — 그래서 아이템 없는 `최백호`는 0이 나온다. `COUNT(*)`로 했으면 1이 나와서 오답. 14강 복습 포인트.

### JOIN 순서

`FROM A JOIN B ON ... JOIN C ON ...` 식으로 위에서 아래로 흘러간다. 일반적으로:

1. **메인 테이블**(결과의 중심이 되는 것)을 `FROM`에
2. 거기서 가까운 관계부터 JOIN을 추가
3. `LEFT JOIN`은 줄지어도 OK, **`INNER JOIN`을 LEFT 뒤에 두면** 거기서 NULL이 탈락하므로 의도와 다를 수 있음

## 8. JOIN 종류 결정 트리

상황이 헷갈리면 이렇게 자문해보자.

```
"양쪽 모두에 데이터가 있는 행만 필요한가?"
   ├─ YES  → INNER JOIN
   └─ NO
       │
       "왼쪽 행은 무조건 다 살리고 싶은가?"
          ├─ YES  → LEFT JOIN
          └─ NO
              │
              "양쪽 모두 살리고 싶은가?"
                 ├─ YES → FULL OUTER JOIN
                 └─ NO  → 다시 생각해보자
```

## 9. 실습 — 다중 JOIN과 SELF JOIN

```sql
-- 1) 모든 유저의 길드명과 아이템 개수
SELECT u.name,
       g.guild_name,
       COUNT(i.id) AS item_count
FROM   users u
LEFT JOIN guilds g  ON g.id = u.guild_id
LEFT JOIN items  i  ON i.user_id = u.id
GROUP BY u.id, u.name, g.guild_name
ORDER BY u.id;

-- 2) 같은 길드에서 나보다 레벨이 높은 사람의 이름
SELECT me.name AS my_name, peer.name AS senior_name, peer.level
FROM users me
JOIN users peer
  ON me.guild_id = peer.guild_id
 AND peer.level > me.level
ORDER BY me.name;

-- 3) 길드별 가입자 수 (가입자 0명인 길드도 표시)
SELECT g.guild_name, COUNT(u.id) AS member_count
FROM   guilds g
LEFT JOIN users u  ON u.guild_id = g.id
GROUP BY g.id, g.guild_name
ORDER BY member_count DESC;
-- → '청룡회'는 0명으로 나옴 (LEFT JOIN 덕분, COUNT(u.id)라 NULL은 안 셈)

-- 4) ⚠️ 함정 재연 — LEFT JOIN인데 우측 조건을 WHERE에 두면
SELECT u.name, i.item_name
FROM users u
LEFT JOIN items i ON i.user_id = u.id
WHERE i.quantity >= 3;
-- → 아이템 없는 유저가 사라짐 (사실상 INNER JOIN)

-- 5) ✅ 올바른 방법 — 우측 조건은 ON에
SELECT u.name, i.item_name
FROM users u
LEFT JOIN items i
  ON i.user_id = u.id
 AND i.quantity >= 3;
-- → 모든 유저 살아남고, 조건 안 맞는 아이템은 NULL
```

## 10. 이번 강의에서 기억할 것

1. **`RIGHT JOIN`은 안 써도 된다.** 테이블 순서만 바꿔서 `LEFT JOIN`으로.
2. **`FULL OUTER JOIN`**은 양쪽 차이 검증 등 특수 용도.
3. **`CROSS JOIN`은 모든 조합.** 쉼표 나열은 실수의 원인 — 명시적으로 쓸 것.
4. **`SELF JOIN`은 같은 테이블을 별칭 다르게 두 번** — 계층/같은 테이블 비교.
5. **`USING (col)`은 양쪽 컬럼명이 같을 때의 단축형.** 기본은 `ON`.
6. **`NATURAL JOIN`은 쓰지 말 것.** 자동 매칭은 위험.
7. **3개 이상 JOIN** 시 행이 곱해질 수 있다 — `COUNT`/`GROUP BY`로 정리하거나 의도 확인.
8. (17강 복습) **`LEFT JOIN`의 우측 조건은 `ON`에**, 좌측 조건은 `WHERE`에.

**19강 — 집합 연산 (`UNION`, `INTERSECT`, `EXCEPT`)**
JOIN은 **가로로 합치는** 연산이라면, 집합 연산은 **세로로 쌓거나 빼는** 연산. "두 테이블의 결과를 한 목록으로", "공통된 것만", "한쪽에만 있는 것만" 같은 쿼리를 다룬다.
