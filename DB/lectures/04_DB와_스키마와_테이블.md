# 4강 — 데이터베이스 / 스키마 / 테이블의 계층 구조

> **목표:** PostgreSQL이 데이터를 보관하는 **3단 계층 구조**를 이해한다.
> **선수 지식:** 1~3강, `psql` 실행 가능 상태

---

## 1. 왜 이걸 따로 배우나?

지난 시간에 무심코 이렇게 했다.

```sql
CREATE DATABASE study;   -- DB를 만들고
\c study                 -- 접속해서
CREATE TABLE test_users (...); -- 테이블을 만들었다
```

여기서 의문이 들어야 한다.

- `study`는 도대체 어디에 만들어진 거지?
- `test_users` 테이블은 `study` 안 어디에 있는 거지?
- 다른 DB로 옮기면 그 테이블은 보이나? **안 보인다. 왜?**

이걸 정리해주는 게 오늘 배울 **계층 구조**다.

---

## 2. PostgreSQL의 3단 구조

```
┌──────────────────────────────────────────────┐
│ PostgreSQL 서버 (인스턴스 1개, 포트 5432)       │
│  ┌────────────────────────────────────────┐  │
│  │ 데이터베이스 (Database)  -- ex) study    │  │
│  │  ┌──────────────────────────────────┐  │  │
│  │  │ 스키마 (Schema)  -- ex) public    │  │  │
│  │  │  ┌───────────────────────────┐   │  │  │
│  │  │  │ 테이블 (Table) -- ex) users│   │  │  │
│  │  │  │ 테이블 -- ex) items        │   │  │  │
│  │  │  │ 테이블 -- ex) guilds       │   │  │  │
│  │  │  └───────────────────────────┘   │  │  │
│  │  └──────────────────────────────────┘  │  │
│  └────────────────────────────────────────┘  │
└──────────────────────────────────────────────┘
```

위에서 아래로 **감싸는 관계**다.

| 계층 | 역할 | C++/언리얼 비유 |
|------|------|----------------|
| **서버 (Instance)** | PostgreSQL 프로그램 본체. 컴퓨터에 1개 설치 | 게임 엔진 |
| **데이터베이스 (Database)** | 완전히 분리된 데이터 공간. 서로 데이터 공유 안 됨 | 프로젝트(.uproject) |
| **스키마 (Schema)** | 테이블·함수를 묶는 폴더 같은 단위 | 폴더 / 네임스페이스 |
| **테이블 (Table)** | 실제 데이터가 들어가는 표 | `UCLASS` / `struct` |

---

## 3. 데이터베이스(Database) 한 단계

### 핵심 성질

- PostgreSQL 서버 안에는 **여러 개의 DB**가 들어갈 수 있다.
- 하지만 **DB끼리는 서로의 데이터를 직접 못 본다.** (강하게 격리됨)
- `psql`에서 한 번에 접속할 수 있는 DB는 **딱 하나.**

### 비유

언리얼에서 프로젝트 A를 열면 프로젝트 B의 블루프린트는 보이지 않는다. 그것과 똑같다.

### 명령어 정리

```sql
-- DB 목록 보기 (psql 메타 명령어)
\l

-- 새 DB 만들기
CREATE DATABASE game_dev;

-- 다른 DB로 접속 바꾸기
\c game_dev

-- DB 지우기 (조심!)
DROP DATABASE game_dev;
```

> 💡 보통 **프로젝트 1개당 DB 1개**를 쓴다. 한 게임 서버는 `game_prod`, 테스트용은 `game_test` 식으로 나누기도.

## 4. 스키마(Schema) — 가장 헷갈리는 개념

여기서 헷갈림 폭발 주의.

### 용어가 두 가지 의미로 쓰인다

1. **(일반 용어) 테이블 구조 설계도** — 2강에서 배운 그 의미.
2. **(PostgreSQL 용어) DB 안에서 테이블들을 묶는 폴더 같은 단위.**

이번 4강에서 말하는 스키마는 **2번** 의미다.

### 스키마는 폴더다

언리얼 콘텐츠 브라우저에 `/Characters/`, `/Weapons/` 폴더를 만들어 블루프린트를 분류하듯,
PostgreSQL은 **스키마**로 테이블들을 분류한다.

```
study (DB)
├─ public (스키마)
│   ├─ users
│   ├─ items
│   └─ guilds
├─ analytics (스키마)
│   ├─ daily_log
│   └─ revenue
└─ archive (스키마)
    └─ old_users
```

### `public` 스키마

- DB를 만들면 **자동으로 `public` 스키마가 같이 생긴다.**
- 우리가 `CREATE TABLE users (...)` 라고만 쓰면, **`public.users`** 로 들어간다.
- 처음 학습할 땐 그냥 `public`만 써도 충분하다.

### 스키마 명령어

```sql
-- 현재 DB의 스키마 목록 (psql)
\dn

-- 스키마 만들기
CREATE SCHEMA analytics;

-- 특정 스키마에 테이블 만들기 (점으로 구분)
CREATE TABLE analytics.daily_log (
    id   INTEGER,
    note TEXT
);

-- 조회할 때도 점 표기로
SELECT * FROM analytics.daily_log;
```

### "왜 굳이 스키마로 나누지?"

- **권한 분리:** 특정 스키마는 특정 유저만 접근하게 막을 수 있음.
- **이름 충돌 방지:** `public.users`와 `archive.users`처럼 같은 이름 가능.
- **정리 정돈:** 테이블이 100개쯤 되면 폴더 분류가 필수.

처음엔 신경 쓰지 말고, **나중에 규모가 커지면 쓰는 도구** 정도로만 기억해두자.

## 5. 테이블(Table) — 데이터의 실제 거주지

이건 2강에서 배운 그대로다. 행과 열로 된 표.

```
study.public.users
┌────┬────────┬───────┬───────┐
│ id │ name   │ level │ gold  │
├────┼────────┼───────┼───────┤
│ 1  │ 김철수  │ 12    │ 500   │
│ 2  │ 이영희  │ 45    │ 12000 │
└────┴────────┴───────┴───────┘
```

테이블의 **전체 이름(Fully Qualified Name)** 은 사실 3단어다.

```text
데이터베이스.스키마.테이블
study.public.users
```

평소엔 다 안 쓰고 `users` 라고만 쓰는데, 그건 PostgreSQL이 "현재 접속한 DB, 기본 스키마"를 자동으로 채워주기 때문이다.

---

## 6. 검색 경로 (search_path) — 잠깐만 짚고 가기

"스키마 이름 안 쓰면 어디서 찾지?"의 답.

PostgreSQL은 **search_path**라는 설정에서 정해진 스키마 목록을 순서대로 뒤져서 테이블을 찾는다. 기본값은 보통 `"$user", public` 인데, 결과적으로 **`public` 스키마가 기본**이 된다.

```sql
-- 현재 search_path 확인
SHOW search_path;
```

지금은 **"public이 기본이다"** 만 기억하면 OK. 5강부터는 쭉 `public`에서 작업한다.

---

## 7. 사용자(Role)도 슬쩍 소개

계층은 아니지만 같이 알아두면 좋다.

- PostgreSQL의 사용자/그룹을 **Role**이라고 부른다.
- 설치 시 만들어진 `postgres` 가 슈퍼 유저 (= 관리자).
- 실무에선 앱 전용 Role을 따로 만들어서 권한을 제한한다.

학습 중엔 그냥 `postgres` 로 다 한다.

## 8. 미니 실습 — 계층 직접 체험

`psql`에서 `study=#` 상태라고 가정.

```sql
-- 1. 새 스키마 만들기
CREATE SCHEMA gamedb;

-- 2. 스키마 목록 확인
\dn

-- 3. 새 스키마에 테이블 만들기
CREATE TABLE gamedb.players (
    id   INTEGER,
    nick TEXT
);

-- 4. 데이터 넣기
INSERT INTO gamedb.players (id, nick) VALUES (1, '용사123');

-- 5. 조회
SELECT * FROM gamedb.players;

-- 6. public 스키마에 같은 이름으로도 만들 수 있다 (충돌 없음!)
CREATE TABLE players (id INTEGER, nick TEXT);
INSERT INTO players (id, nick) VALUES (1, '마법사456');

-- 7. 둘 다 따로 존재함을 확인
SELECT * FROM gamedb.players;   -- 용사123
SELECT * FROM public.players;   -- 마법사456
SELECT * FROM players;          -- search_path가 public 우선이므로 마법사456
```

이 실습을 한 번 직접 쳐보면 **"스키마는 정말 폴더 같구나"** 라는 감이 온다.

## 9. 이번 강의에서 기억할 것

1. PostgreSQL의 계층: **서버 > 데이터베이스 > 스키마 > 테이블.**
2. **DB끼리는 격리**된다. 한 번에 한 DB만 접속.
3. **스키마는 폴더**처럼 테이블을 묶는 단위. 기본은 `public`.
4. 테이블의 전체 이름은 `데이터베이스.스키마.테이블`.
5. **"스키마"라는 단어는 두 가지 뜻**이 있다 — 설계도 vs 폴더 단위. 문맥으로 구분.

---

## 10. 다음 강의 예고 — 2단계 시작

**5강 — `CREATE TABLE`로 테이블 만들기**
드디어 본격적인 **DDL(Data Definition Language)** 시작. `CREATE TABLE` 문법을 제대로 익히고, 데이터 타입에는 어떤 게 있는지 둘러본다. 6강에서 타입을 더 깊이 파고든다.

---

## 11. 1단계 마무리 정리

이번 강의로 **1단계(기초 개념)** 가 끝난다. 여기까지 잡고 가는 큰 그림.

| 강의 | 핵심 한 줄 |
|------|----------|
| 1강 | DB는 데이터를 안전·효율적으로 관리하는 도구. PostgreSQL은 DBMS. |
| 2강 | RDBMS는 데이터를 표(테이블)로 저장. 행=한 건, 열=한 종류. |
| 3강 | `psql`로 DB와 대화. 세미콜론 필수. 메타 명령어는 `\`. |
| 4강 | DB > 스키마 > 테이블의 3단 구조. 기본 스키마는 `public`. |

이 정도 깔리면 다음 단계는 훨씬 부드럽다. 다음 강의부터 본격적으로 SQL을 쓴다.
