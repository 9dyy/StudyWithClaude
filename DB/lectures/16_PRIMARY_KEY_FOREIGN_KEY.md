# 16강 — `PRIMARY KEY`와 `FOREIGN KEY`

> **목표:** 행을 **유일하게 식별**하고, 테이블 간 **관계를 안전하게** 연결한다.
> **선수 지식:** 5~9강 (특히 `CREATE TABLE`, `ALTER TABLE`)

---

## 1. 왜 이게 필요한가?

5강에서 만들었던 테이블을 다시 보자.

```sql
CREATE TABLE users (
    id        INTEGER,
    name      TEXT,
    level     INTEGER,
    gold      INTEGER,
    guild_id  INTEGER     -- guilds.id를 가리킬 예정
);
```

문제 두 가지:

1. **유저 두 명이 같은 `id`를 가지면?** — 누가 누군지 구분 못 한다.
2. **`guild_id = 99`인데 길드 테이블엔 99번이 없으면?** — 유령 길드를 가리키는 깨진 데이터.

이걸 막아주는 두 제약이 오늘 주인공.

| 제약 | 역할 | 한 줄 |
|------|------|-------|
| `PRIMARY KEY` (PK) | **유일 식별자** | "이 행은 나 하나뿐이다" |
| `FOREIGN KEY` (FK) | **참조 무결성** | "내가 가리키는 그 행, 반드시 존재한다" |

C++로 비유하면 PK는 **객체의 고유 ID(포인터 주소 같은 것)**, FK는 **다른 객체를 가리키는 포인터(단, dangling은 절대 금지)**.

---

## 2. `PRIMARY KEY` — 기본 키

### 개념

테이블의 **한 행을 유일하게 식별**하는 컬럼(들). 다음을 자동으로 강제한다:

- **`UNIQUE`** — 같은 값 중복 금지
- **`NOT NULL`** — NULL 금지
- **인덱스 자동 생성** — 검색이 빠름

테이블당 **PK는 단 하나**. (UNIQUE는 여러 개 가능, PK는 1개.)

### 문법 1 — 컬럼 옆에 인라인으로

```sql
CREATE TABLE users (
    id    INTEGER PRIMARY KEY,
    name  TEXT NOT NULL
);
```

### 문법 2 — 테이블 정의 마지막에 따로

```sql
CREATE TABLE users (
    id    INTEGER,
    name  TEXT NOT NULL,
    CONSTRAINT pk_users PRIMARY KEY (id)
);
```

두 방식의 결과는 같다. **`CONSTRAINT 이름 ...`** 형태가 실무에서 권장된다. 이름을 안 붙이면 PostgreSQL이 자동 이름(예: `users_pkey`)을 만들어주는데, 나중에 에러 메시지나 마이그레이션에서 **제약 이름이 명시적**일수록 디버깅이 훨씬 편하다.

### 자동증가 PK — `GENERATED BY DEFAULT AS IDENTITY`

매번 `id`를 직접 박는 건 귀찮다. PostgreSQL이 자동으로 부여하게 하자.

```sql
CREATE TABLE users (
    id    BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name  TEXT NOT NULL
);

INSERT INTO users (name) VALUES ('김철수');   -- id 자동 부여 (1)
INSERT INTO users (name) VALUES ('이영희');   -- id 자동 부여 (2)
```

`BY DEFAULT` vs `ALWAYS`:

| 모드 | 직접 값 주면? |
|------|--------------|
| `GENERATED BY DEFAULT AS IDENTITY` | **덮어쓰기 허용** (`INSERT INTO ... (id) VALUES (100)` OK) |
| `GENERATED ALWAYS AS IDENTITY` | **거부**, 자동 생성만 강제 |

> 💡 보통은 `BY DEFAULT`가 무난. **특수 의미를 가진 ID(예: `id=0`을 "전체" 의미로)를 직접 박을 일**이 생길 수 있고, 데이터 이관 시 기존 id를 그대로 가져와야 할 때도 유용. `ALWAYS`는 "절대 누구도 id를 손대지 못하게" 락을 걸고 싶을 때.

옛날 PostgreSQL의 `SERIAL` / `BIGSERIAL`도 같은 역할을 하지만 **SQL 표준이 아니라** 신규 코드엔 `IDENTITY`를 권장한다.

### 복합 PK — 여러 컬럼을 묶어 식별

한 컬럼으로 식별이 안 될 때. **N:N(다대다) 관계를 푸는 중간 테이블**이 대표적이다.

예) 유저가 여러 아이템을 가지고, 한 아이템 종류를 여러 유저가 가질 수 있을 때 — 중간 테이블 `user_items`로 푼다.

```sql
CREATE TABLE user_items (
    user_id    BIGINT NOT NULL,
    item_id    BIGINT NOT NULL,
    quantity   INTEGER NOT NULL DEFAULT 1,
    created_at TIMESTAMP NOT NULL DEFAULT NOW(),
    CONSTRAINT pk_user_items PRIMARY KEY (user_id, item_id)
);
```

`(user_id, item_id)` 쌍이 유일. 같은 쌍이 두 번 들어올 수 없다 (= 같은 유저가 같은 아이템 종류 행을 두 번 못 만듦, 수량은 `quantity`로 관리).

> 💡 실무에선 **인공 `id` PK + `UNIQUE (user_id, item_id)`** 조합도 자주 쓴다. PK를 단순(단일 BIGINT)하게 유지하면서 같은 효과를 얻는 패턴.

## 3. `FOREIGN KEY` — 외래 키

### 개념

**다른 테이블의 PK(또는 UNIQUE 컬럼)를 가리키는 컬럼.** DB가 "그 값이 진짜 존재하는지" 자동으로 검사한다.

```
guilds.id  ← (참조됨)
   ↑
   │ FK
   │
users.guild_id  (참조함)
```

FK가 있으면 다음이 자동 보장된다:

1. **존재하지 않는 부모 참조 금지** — `guild_id = 999`인데 `guilds`에 999 없으면 INSERT/UPDATE 거부
2. **참조되는 부모는 함부로 못 지움** — 자식이 있는 부모를 그냥 `DELETE`하면 거부 (또는 `ON DELETE` 옵션에 따라 동작)

### 문법

```sql
-- 자식 테이블
CREATE TABLE users (
    id        BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name      TEXT NOT NULL,
    guild_id  BIGINT,
    CONSTRAINT fk_users_guild
      FOREIGN KEY (guild_id) REFERENCES guilds(id)
);
```

읽는 법: "`users.guild_id`는 `guilds.id`를 외래 참조한다."

`REFERENCES guilds(id)`에서 **참조 대상 컬럼은 PK이거나 UNIQUE**여야 한다 (그래야 유일하게 찾을 수 있으니까).

### 나중에 추가하기 — `ALTER TABLE`

테이블을 다 만든 뒤에 FK만 따로 거는 것도 매우 흔하다. **"테이블 먼저, FK 나중에"** 패턴.

```sql
ALTER TABLE users
ADD CONSTRAINT fk_users_guild
FOREIGN KEY (guild_id) REFERENCES guilds(id);
```

왜 그렇게 하느냐? **테이블끼리 서로 참조**할 때 (A → B → A) 한 번에 `CREATE`하면 "아직 없는 테이블 참조" 에러가 난다. 그래서 init 스크립트는 보통:

1. 모든 테이블 `CREATE` (FK 없이)
2. 모든 FK를 `ALTER TABLE ... ADD CONSTRAINT`로 추가

### FK 시연

```sql
-- 길드 미리 2개
INSERT INTO guilds (id, guild_name) VALUES (1, '흑사자단'), (2, '백호단');

-- 정상
INSERT INTO users (name, guild_id) VALUES ('김철수', 1);   -- ✅

-- 존재하지 않는 길드
INSERT INTO users (name, guild_id) VALUES ('이영희', 99);  -- ❌
-- ERROR: insert or update on table "users" violates foreign key constraint "fk_users_guild"

-- guild_id NULL은 허용 (FK 컬럼이 NOT NULL이 아니라면)
INSERT INTO users (name, guild_id) VALUES ('박무소속', NULL);   -- ✅
```

> 💡 **FK는 NULL을 허용한다** (컬럼 자체에 `NOT NULL`이 없다면). "아직 소속 없음"을 NULL로 표현하는 게 자연스러운 경우가 많다. 무소속 유저, 미배정 담당자 등.

## 4. `ON DELETE` / `ON UPDATE` 옵션

부모가 **삭제되거나 PK가 변경**될 때 자식을 어떻게 할 것인가? 5가지 모드가 있다.

| 옵션 | 부모 삭제 시 자식 | 언제 쓰나 |
|------|-------------------|----------|
| `NO ACTION` (기본값) | **거부** (자식 있으면 부모 못 지움) | "안전 잠금" — 실수 방지 |
| `RESTRICT` | 거부 (즉시 검사) | NO ACTION과 거의 동일 |
| `CASCADE` | **같이 삭제** | 자식이 부모 없이 의미 없을 때 |
| `SET NULL` | FK 컬럼을 **NULL로** | 자식 보존, 관계만 끊을 때 |
| `SET DEFAULT` | FK 컬럼을 **DEFAULT 값으로** | 거의 안 씀 |

`ON UPDATE`도 같은 5개 옵션. 보통 `ON UPDATE CASCADE`를 많이 쓴다 (부모 PK가 바뀌면 자식도 따라감 — 사실 IDENTITY PK는 변할 일이 드물어서 무의미한 경우가 많지만 관습적으로 붙임).

### 문법

```sql
ALTER TABLE users
ADD CONSTRAINT fk_users_guild
FOREIGN KEY (guild_id) REFERENCES guilds(id)
ON DELETE SET NULL
ON UPDATE CASCADE;
```

### 어떤 옵션을 고를까?

게임 DB로 예를 들어보자.

| 관계 | 권장 | 이유 |
|------|------|------|
| `users.guild_id → guilds.id` | `SET NULL` | 길드 해체돼도 유저는 살아남음 (무소속이 됨) |
| `items.user_id → users.id` | `CASCADE` | 유저 탈퇴 시 그 유저의 인벤토리 아이템도 같이 정리 |
| `user_items.user_id → users.id` | `CASCADE` | 부모 유저 없으면 소유 관계 무의미 |
| `chat_message.user_id → users.id` | `SET NULL` | 유저 탈퇴해도 **채팅 기록은 보존** (작성자만 NULL) |
| `quest_log.user_id → users.id` | `SET NULL` | 감사/통계용 로그는 유저가 사라져도 보존 |

원칙:
- **자식이 부모 없이 의미 없으면** → `CASCADE`
- **자식은 살아남아야 하고 관계만 끊으면** → `SET NULL` (이 경우 자식 FK 컬럼은 **NOT NULL이 아니어야** 함)
- **함부로 못 지우게 막아야 하면** → `NO ACTION` (기본값)

> ⚠️ `CASCADE`는 강력한 만큼 위험하다. "기업 하나 지웠더니 유저 1만명·프로젝트 5만개가 줄줄이 삭제"가 가능. 정책으로 정해서 신중하게 걸자.

## 5. 자기 참조 FK

한 테이블이 자기 자신을 가리키는 경우. **계층 구조**(트리)를 표현할 때 쓴다.

```sql
CREATE TABLE categories (
    id        BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name      TEXT NOT NULL,
    parent_id BIGINT,
    CONSTRAINT fk_category_parent
      FOREIGN KEY (parent_id) REFERENCES categories(id) ON DELETE CASCADE
);
```

- 최상위 카테고리는 `parent_id = NULL`
- 하위 카테고리는 부모의 `id`를 가리킴

```
categories
├─ '게임' (id=1, parent_id=NULL)
│   ├─ 'RPG' (id=2, parent_id=1)
│   └─ 'FPS' (id=3, parent_id=1)
└─ '음악' (id=4, parent_id=NULL)
```

C++의 **부모-자식 트리 노드**와 같은 구조. 재귀 쿼리(`WITH RECURSIVE`)로 탐색한다 — 28강에서 다룰 예정.

## 6. PK / FK / UNIQUE 비교

헷갈리기 쉬운 셋을 한 번에.

| 제약 | NULL 허용? | 중복 허용? | 테이블당 개수 | 자동 인덱스 |
|------|-----------|-----------|--------------|------------|
| `PRIMARY KEY` | ❌ | ❌ | **1개** | ✅ |
| `UNIQUE` | ✅ (NULL은 중복으로 보지 않음) | ❌ | 여러 개 가능 | ✅ |
| `FOREIGN KEY` | ✅ (컬럼이 NOT NULL이 아니면) | ✅ (자식 입장에선 중복 가능) | 여러 개 가능 | ❌ (직접 만들어야 함!) |

> ⚠️ **FK 컬럼엔 인덱스가 자동으로 안 생긴다.** JOIN 성능과 부모 삭제 시 자식 검사를 위해 거의 항상 **수동으로 인덱스를 걸어야** 한다. 29강(인덱스)에서 자세히.

## 7. 실습 — 5강 테이블에 PK/FK 입히기

5강에서 만들었던 테이블들(`guilds`, `users`, `items`)은 일부러 제약 없이 만들었다. 이제 제대로 만들어보자.

```sql
-- 깔끔하게 새로 시작
DROP TABLE IF EXISTS items;
DROP TABLE IF EXISTS users;
DROP TABLE IF EXISTS guilds;

-- 1) guilds — 부모 테이블 먼저
CREATE TABLE guilds (
    id         BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    guild_name TEXT NOT NULL UNIQUE
);

-- 2) users — guilds 참조
CREATE TABLE users (
    id        BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name      TEXT NOT NULL,
    level     INTEGER NOT NULL DEFAULT 1,
    coin      INTEGER NOT NULL DEFAULT 0,
    guild_id  BIGINT,
    CONSTRAINT fk_users_guild
      FOREIGN KEY (guild_id) REFERENCES guilds(id)
      ON DELETE SET NULL ON UPDATE CASCADE
);

-- 3) items — users 참조
CREATE TABLE items (
    id        BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id   BIGINT NOT NULL,
    item_name TEXT NOT NULL,
    quantity  INTEGER NOT NULL DEFAULT 1 CHECK (quantity > 0),
    CONSTRAINT fk_items_user
      FOREIGN KEY (user_id) REFERENCES users(id)
      ON DELETE CASCADE ON UPDATE CASCADE
);

-- 데이터 넣어보기
INSERT INTO guilds (guild_name) VALUES ('흑사자단'), ('백호단');

INSERT INTO users (name, level, coin, guild_id)
VALUES ('김철수', 30, 1500, 1),
       ('이영희', 45, 3200, 1),
       ('박무소속', 12, 100, NULL);

INSERT INTO items (user_id, item_name, quantity)
VALUES (1, '회복포션', 5),
       (1, '강철검', 1),
       (2, '마법서', 3);

-- 제약 위반 시도들
INSERT INTO users (name, guild_id) VALUES ('유령', 99);  -- ❌ guilds에 99 없음
INSERT INTO items (user_id, item_name) VALUES (99, '?'); -- ❌ users에 99 없음

-- ON DELETE 동작 확인
DELETE FROM guilds WHERE id = 1;
-- 흑사자단 소속 유저들의 guild_id가 NULL로 (SET NULL 동작)
SELECT name, guild_id FROM users;

DELETE FROM users WHERE id = 2;
-- 이영희의 아이템들이 같이 사라짐 (CASCADE 동작)
SELECT * FROM items;
```

> 💡 **위에서 아래로 부모→자식 순서로 만들었다.** FK는 참조 대상이 먼저 존재해야 걸 수 있다. 반대로 **지울 땐 자식부터** (`DROP TABLE items, users, guilds` 순) — `CASCADE`를 안 썼다면.

## 8. 이미 만든 테이블에 PK/FK 나중에 붙이기

운영 중인 DB에 자주 일어나는 일.

```sql
-- PK가 없던 테이블에 부여
ALTER TABLE old_table ADD CONSTRAINT pk_old_table PRIMARY KEY (id);

-- FK 추가
ALTER TABLE users
ADD CONSTRAINT fk_users_guild
FOREIGN KEY (guild_id) REFERENCES guilds(id)
ON DELETE SET NULL;

-- 제거
ALTER TABLE users DROP CONSTRAINT fk_users_guild;
```

> ⚠️ **기존 데이터가 제약을 위반하면 추가가 실패한다.** PK 부여 전엔 중복/NULL부터, FK 부여 전엔 **고아 행**(부모 없는 자식)부터 정리해야 한다.
>
> ```sql
> -- 고아 행 찾기
> SELECT * FROM users u
> WHERE u.guild_id IS NOT NULL
>   AND NOT EXISTS (SELECT 1 FROM guilds g WHERE g.id = u.guild_id);
> ```
> (`NOT EXISTS`는 25강에서 자세히)

## 9. 자주 하는 실수

1. **FK 컬럼과 참조 대상의 타입 불일치**
   `guild_id INTEGER` vs `guilds.id BIGINT` → FK 생성 실패. **타입을 똑같이.**

2. **FK 컬럼에 인덱스 안 거는 것**
   JOIN이 느려지고, 부모 삭제 시 자식 검사가 풀스캔 된다. **거의 항상 인덱스 같이.**

3. **`CASCADE` 남발**
   "기업 하나 지웠더니 모든 게 사라졌어요" 사고. **데이터 보존이 필요한 자식은 `SET NULL` 또는 `NO ACTION`.**

4. **PK 없이 운영하는 테이블**
   복제·동기화·UPDATE 시 어떤 행을 가리키는지 모호해진다. **모든 테이블에 PK는 거의 필수.**

5. **제약 이름 지정 안 함**
   `CONSTRAINT pk_xxx` 없이 인라인으로 쓰면 PostgreSQL이 자동 이름(예: `users_pkey`)을 만든다. 동작은 하지만 **나중에 마이그레이션·디버깅 때 불편**. 실무에선 명시적 이름 권장.

## 10. 이번 강의에서 기억할 것

1. **`PRIMARY KEY`** = 유일 식별자. `UNIQUE` + `NOT NULL` + 자동 인덱스, 테이블당 1개.
2. **자동증가 PK는 `BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`**. 신규 코드 표준.
3. **`FOREIGN KEY`** = 다른 테이블 PK를 가리킴. **참조 무결성**을 DB가 강제.
4. **`ON DELETE` 5종**: `NO ACTION`(기본·거부), `CASCADE`(같이 삭제), `SET NULL`(관계만 끊음). 정책에 맞춰 신중히.
5. **"테이블 먼저, FK 나중에"** — 큰 init 스크립트의 표준 순서.
6. **FK 컬럼엔 인덱스가 자동으로 안 붙는다** — 거의 항상 수동으로 거는 게 정석.
7. **제약엔 이름을 붙이자** (`CONSTRAINT fk_xxx ...`) — 디버깅·마이그레이션이 훨씬 편해진다.

## 11. 다음 강의 예고

**17강 — `JOIN` 기본**
드디어 **두 테이블을 묶어서** 조회한다. "유저 이름과 그 유저의 길드 이름을 한 번에" 같은 쿼리가 비로소 가능해진다. PK/FK가 진짜 빛을 발하는 순간. `INNER JOIN`, `LEFT JOIN`, `ON`/`USING` 절을 차근차근.
