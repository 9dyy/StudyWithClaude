# 2강 — 커널 모드와 유저 모드, 시스템 콜

> **목표:** 우리가 짠 게임 코드가 OS 기능을 부를 때 CPU 안에서 실제로 무슨 일이 일어나는지 이해한다.
> **선수 지식:** 1강 (OS는 자원 관리자 + 하드웨어 추상화).

---

## 1. 도입 — 왜 굳이 "모드"라는 게 있을까?

게임을 짜다가 이런 코드를 보면 어떨까.

```cpp
void Hack()
{
    // 다른 프로세스(예: 카카오톡)의 메모리 영역을 통째로 읽어버린다
    char* OtherProcessMemory = reinterpret_cast<char*>(0x12345678);
    char Stolen = *OtherProcessMemory;
}
```

이게 그냥 되면 큰일이다. 내 게임이 카톡 메시지를 훔쳐보고, 크롬에 저장된 비밀번호를 빼낼 수 있다는 뜻이니까.

또 이런 것도 안 된다.

```cpp
// SSD 컨트롤러에 직접 명령 보내서 다른 프로세스 파일 다 지우기
SendNvmeCommand(0xDEAD, ERASE_ALL);
```

이런 짓을 못 하게 막는 장치가 CPU 차원에서 있다. 그게 **유저 모드 / 커널 모드** 다.

---

## 2. 두 가지 모드

CPU에는 **권한 레벨(Privilege Level)** 이라는 게 있다. 인텔/AMD CPU는 ring 0 ~ ring 3 까지 4단계가 있는데, 현대 OS는 보통 두 단계만 쓴다.

| 모드 | Ring | 누가 쓰나 | 권한 |
|------|------|----------|------|
| **유저 모드** | ring 3 | 우리 게임, 크롬, 디스코드 등 | 자기 메모리만 접근 가능. 하드웨어 직접 못 만짐. 특권 명령 못 씀. |
| **커널 모드** | ring 0 | OS 커널, 디바이스 드라이버 | 모든 메모리, 모든 하드웨어, 모든 명령 접근 가능. |

우리 게임(`.exe`)은 **항상 유저 모드**에서 돈다. SSD에 직접 명령? 못 한다. 다른 프로세스 메모리? 못 본다. 그래서 위에서 본 `Hack()` 같은 코드는 **CPU가 거부**한다 (Access Violation, segfault).

> **비유:** 회사 사무실에 비유하면
> - 유저 모드 = 일반 직원. 자기 자리에서만 일함. 서버실 못 들어감.
> - 커널 모드 = 관리자. 서버실 들어가고, 다른 자리도 둘러볼 수 있음.

---

## 3. 그럼 게임이 파일을 어떻게 읽지?

게임은 분명 SSD에서 파일을 읽는다. 그런데 유저 모드라서 SSD를 직접 못 만진다. 어떻게?

**답: OS한테 "대신 해줘" 라고 부탁한다. 이게 시스템 콜(System Call).**

```cpp
// 우리 코드 (유저 모드)
FString Text;
FFileHelper::LoadFileToString(Text, TEXT("Config.ini"));
```

이 한 줄 안에서 벌어지는 일:

```
[유저 모드]                                [커널 모드]
  우리 코드
    │
    │ LoadFileToString()
    ↓
  CRT의 fopen()
    │
    ↓
  Win32 API의 ReadFile()
    │
    │ syscall 명령 실행 ─────────────────→ 커널 진입 (Mode Switch!)
                                              │
                                              ↓
                                            NtReadFile() (커널 함수)
                                              │
                                              ↓
                                            파일 시스템 드라이버
                                              │
                                              ↓
                                            SSD 컨트롤러에 명령
                                              │
                                              ↓
                                            결과 받기
                                              │
    ┌─── 결과 복사 ←─────────────────────── 유저 모드로 복귀 (Mode Switch!)
    ↓
  Text 변수에 값 들어옴
```

핵심 포인트:

1. **유저 모드 → 커널 모드로 넘어가는 순간이 있다.** 이게 시스템 콜.
2. 넘어가는 명령어는 CPU 명령어 하나다 (x86은 `syscall` 또는 `sysenter`).
3. 커널 일이 끝나면 다시 유저 모드로 돌아온다.

---

## 4. 시스템 콜은 비싸다

이 모드 전환(context switch이 아니라 **mode switch**)이 공짜가 아니다.

- CPU 레지스터 일부를 저장/복원해야 함
- 커널 스택으로 바꿔야 함
- 캐시/TLB 일부 무효화 가능
- 보안 검사 (어떤 인자가 들어왔는지 확인)

대략 **수백 ~ 수천 사이클** 정도. 함수 호출 한 번이 보통 수 사이클인 것과 비교하면 어마어마하게 비싸다.

### 게임에서 이게 왜 중요한가

```cpp
// (1) 안 좋은 예 — 매 프레임 디스크에서 작은 파일 1만 번 읽기
for (int i = 0; i < 10000; ++i)
{
    FString S;
    FFileHelper::LoadFileToString(S, FilePath);   // 시스템 콜 × 10000
}

// (2) 좋은 예 — 한 번에 다 읽고 메모리에서 처리
TArray<uint8> Buffer;
FFileHelper::LoadFileToArray(Buffer, FilePath);   // 시스템 콜 1번
ParseInMemory(Buffer);                            // 시스템 콜 없음
```

엔진들이 **풀 할당자(Pool Allocator)**, **배치 처리**, **버퍼링**을 쓰는 이유 중 하나가 이거다. 시스템 콜 횟수를 줄이려고.

언리얼의 `UE_LOG`도 디버그 빌드에서 미친 듯이 찍으면 게임이 느려지는데, 콘솔 출력이 시스템 콜이라 그렇다.

---

## 5. 시스템 콜의 종류 (대표)

| 분류 | 예시 (Win32 / POSIX) | 게임에서 |
|------|---------------------|----------|
| 프로세스 | `CreateProcess` / `fork`, `exec` | 게임에서 외부 프로세스 띄울 때 (런처) |
| 스레드 | `CreateThread` / `pthread_create` | 비동기 작업 |
| 메모리 | `VirtualAlloc` / `mmap`, `brk` | 큰 메모리 할당 시 |
| 파일 | `CreateFile`, `ReadFile`, `WriteFile` | 에셋 로딩, 세이브 |
| 네트워크 | `socket`, `send`, `recv` | 멀티플레이 통신 |
| 동기화 | `CreateMutex`, `WaitForSingleObject` | 스레드 동기화 |
| 시간 | `QueryPerformanceCounter` | 프레임 타임 측정 |

언리얼의 `FPlatformMemory::Malloc`, `FPlatformFile`, `FRunnableThread`, `FCriticalSection` 같은 게 다 내부적으로 이걸 부른다.

## 6. 코드로 확인하기 — 같은 일을 한 번 vs 여러 번 했을 때

언리얼에서 직접 시간 재보면 차이를 느낄 수 있다.

```cpp
void DemoSyscallCost()
{
    const int N = 100000;
    const FString FilePath = FPaths::ProjectDir() / TEXT("test.txt");

    // 한 번에 큰 거 쓰기 (시스템 콜 1번)
    {
        FString Big;
        Big.Reserve(N * 10);
        for (int i = 0; i < N; ++i) Big += TEXT("Hello\n");

        double Start = FPlatformTime::Seconds();
        FFileHelper::SaveStringToFile(Big, *FilePath);
        UE_LOG(LogTemp, Log, TEXT("Batch: %.3f ms"),
               (FPlatformTime::Seconds() - Start) * 1000.0);
    }

    // 작은 거 N번 쓰기 (시스템 콜 N번)
    {
        double Start = FPlatformTime::Seconds();
        for (int i = 0; i < N; ++i)
        {
            FFileHelper::SaveStringToFile(TEXT("Hello\n"), *FilePath,
                FFileHelper::EEncodingOptions::AutoDetect,
                &IFileManager::Get(),
                FILEWRITE_Append);
        }
        UE_LOG(LogTemp, Log, TEXT("Per-write: %.3f ms"),
               (FPlatformTime::Seconds() - Start) * 1000.0);
    }
}
```

대개 후자가 수십~수백 배 느리다. 디스크 캐시까지 감안해도 시스템 콜 자체가 누적되는 게 크다.

## 7. 흔한 오해

### 오해 1: "함수 호출 = 시스템 콜"

아니다. **유저 모드 안에서 끝나는 함수 호출은 시스템 콜이 아니다.** 예:

```cpp
int a = MyAdd(1, 2);     // 그냥 함수 호출. mode switch 없음.
FMath::Sqrt(16.0f);      // 마찬가지.
TArray<int> Arr;         // 생성자 호출. 메모리 할당이 없으면 syscall 없음.
```

반면:

```cpp
FRunnableThread::Create(...);  // 내부에서 CreateThread → syscall 있음
FFileHelper::LoadFileToString(...); // 내부에서 ReadFile → syscall 있음
```

### 오해 2: "커널 모드 = 빠름, 유저 모드 = 느림"

전혀 아니다. **권한의 문제일 뿐이지 속도 자체는 같다.** 다만 모드를 **바꾸는 비용**이 비싼 거다.

### 오해 3: "디버거가 다른 프로세스 메모리를 보는 건 어떻게?"

디버거도 유저 모드 프로그램이다. 하지만 `OpenProcess`, `ReadProcessMemory` 같은 **특수 시스템 콜**을 부르고, OS가 "이 프로세스는 디버그 권한이 있군" 하고 허락해주는 거다. 결국 통로는 시스템 콜 하나뿐.

## 8. 면접에서 나오면

**Q1. 유저 모드와 커널 모드의 차이는?**
→ "CPU의 권한 레벨 차이입니다. 유저 모드(ring 3)는 자기 프로세스 메모리만 접근 가능하고 특권 명령을 쓸 수 없습니다. 커널 모드(ring 0)는 모든 메모리와 하드웨어에 접근할 수 있습니다. 응용 프로그램은 유저 모드에서, OS 커널과 드라이버는 커널 모드에서 돕니다."

**Q2. 시스템 콜이 뭔가요? 왜 비싼가요?**
→ "유저 모드 프로그램이 커널 기능을 호출하는 통로입니다. `syscall` 명령으로 커널 모드로 진입하는데, 이때 CPU 레지스터를 저장하고 커널 스택으로 전환하고 인자 검증을 거치기 때문에 일반 함수 호출보다 수백~수천 배 비쌉니다. 그래서 게임 엔진은 시스템 콜 횟수를 줄이려고 메모리 풀이나 배치 I/O를 씁니다."

**Q3. 일반 함수 호출과 시스템 콜은 뭐가 다른가요?**
→ "일반 함수 호출은 유저 모드 안에서 끝나서 모드 전환이 없습니다. 시스템 콜은 유저 모드 → 커널 모드로 넘어갔다 돌아오는 모드 스위치가 일어납니다."

**Q4. 게임 코드에서 다른 프로세스 메모리를 직접 읽을 수 있나요?**
→ "유저 모드라서 직접은 못 합니다. CPU가 차단해서 Access Violation이 납니다. 다만 `ReadProcessMemory` 같은 시스템 콜을 통해 OS에 부탁할 수 있고, 권한이 있으면 OS가 대신 읽어줍니다."

## 9. 셀프 체크

- [ ] 유저 모드와 커널 모드가 CPU의 권한 레벨이라는 걸 안다.
- [ ] 시스템 콜이 모드 스위치를 동반하는 비싼 호출이라는 걸 안다.
- [ ] 게임에서 `for`문 안에 `LoadFileToString` 만 번 부르면 왜 느린지 설명할 수 있다.
- [ ] "유저 모드 함수 호출"과 "시스템 콜"을 구분할 수 있다.
