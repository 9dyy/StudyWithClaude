# 3강 — 프로세스의 메모리 구조

> **목표:** 내가 짠 `.exe`가 실행되면 메모리에 어떻게 배치되는지 이해한다. 코드/데이터/힙/스택의 정체.
> **선수 지식:** 1~2강. C++의 `new`, 지역 변수, 전역 변수 정도는 써봤어야 함.

---

## 1. 도입 — `.exe` 더블클릭하면 무슨 일이?

`MyGame.exe`를 더블클릭한다.

```
1. OS가 MyGame.exe 파일을 디스크에서 읽는다.
2. OS가 메모리(RAM)에 영역을 잡는다.
3. exe 안에 있던 "코드"와 "데이터"를 그 메모리에 배치한다.
4. main() 함수의 주소로 점프해서 실행 시작.
```

이렇게 "실행 중인 프로그램"이 된 게 **프로세스(Process)** 다.

> `.exe` 파일 = 디스크에 있는 정적인 바이트
> 프로세스 = 메모리에 올라와서 돌고 있는 실체

같은 `MyGame.exe`를 2번 더블클릭하면 **프로세스가 2개** 생긴다. 각자 별도 메모리 공간을 가진다.

---

## 2. 프로세스의 메모리 레이아웃

프로세스 하나가 받는 메모리는 대략 이렇게 생겼다. (32비트 기준으로 그리면 깔끔해서 32비트로)

```
높은 주소 (0xFFFFFFFF)
┌──────────────────────┐
│   커널 영역           │  ← 내 프로세스가 못 건드림
├──────────────────────┤
│   스택 (Stack)        │  ← 함수 호출, 지역 변수. 위에서 아래로 자람
│         ↓             │
│                       │
│   (빈 공간)           │
│                       │
│         ↑             │
│   힙 (Heap)           │  ← new, malloc. 아래에서 위로 자람
├──────────────────────┤
│   BSS                │  ← 초기화 안 된 전역/static 변수 (0으로 채워짐)
├──────────────────────┤
│   Data               │  ← 초기화된 전역/static 변수
├──────────────────────┤
│   Text (Code)         │  ← 기계어 코드. 읽기 전용.
└──────────────────────┘
낮은 주소 (0x00000000)
```

영역별로 뜯어보자.

### Text (Code) 영역

내가 짠 함수들이 컴파일되어 만들어진 **기계어**가 여기 있다.

```cpp
void Foo() { ... }     // 이 함수의 기계어 → Text 영역
int main() { ... }     // 이 함수의 기계어 → Text 영역
```

특징:
- **읽기 전용**. 실행 중에 코드를 덮어쓰면 OS가 죽인다.
- 같은 `.exe`를 2번 띄우면, 영리한 OS는 Text 영역을 공유해서 RAM을 아낀다.

### Data 영역

초기값이 있는 **전역/static 변수**.

```cpp
int g_Score = 100;          // Data 영역
static int s_Count = 0;     // Data 영역 (0도 명시적이면 보통 BSS 가지만 처리 방식은 컴파일러마다 다름)
```

### BSS 영역

초기화 안 된 전역/static 변수. 프로그램 시작 시 **OS가 0으로 채워준다.**

```cpp
int g_Buffer[1024];   // BSS — 자동으로 전부 0
static char s_Name[256];  // BSS
```

> 왜 분리되어 있나? Data는 exe 파일에 실제 초기값이 박혀 있어서 용량을 잡아먹지만, BSS는 "크기만 알려줘, 어차피 0이니까" 하고 exe에 거의 공간을 안 잡아먹는다. 1MB짜리 배열을 전역으로 선언해도 exe는 안 커지는 이유.

### Heap 영역

`new`, `malloc`으로 동적 할당하는 메모리.

```cpp
int* P = new int[1000];           // Heap에 4000바이트 잡음
TArray<int32> Arr;
Arr.SetNum(1000);                 // 내부적으로 Heap에 할당
UObject* Obj = NewObject<UMyObj>(); // UObject도 결국 Heap에서 할당
```

특징:
- **아래에서 위로 자란다** (주소가 커지는 방향).
- 할당/해제 시 OS의 메모리 관리자가 개입. 비싸다.
- **수명을 내가 직접 관리해야 함.** 안 풀면 메모리 누수.

### Stack 영역

함수 호출 시 만들어지는 **지역 변수**, **함수 인자**, **리턴 주소** 가 들어간다.

```cpp
void Foo(int Arg)              // Arg → Stack
{
    int Local = 42;            // Local → Stack
    FVector V(1, 2, 3);        // V → Stack
}
```

특징:
- **위에서 아래로 자란다** (주소가 작아지는 방향).
- 함수 호출 시 스택 프레임 push, 리턴 시 pop. **공짜**(레지스터 하나만 움직임).
- 크기 제한 있음 (보통 1~8MB). 넘으면 **스택 오버플로**.

```cpp
void StackOverflow()
{
    char Big[10 * 1024 * 1024];   // 10MB짜리 지역 변수
    // 스택 한도 넘어서 크래시
}

void RecursionOverflow()
{
    RecursionOverflow();   // 무한 재귀 → 스택 프레임 무한 쌓임 → 크래시
}
```

## 3. 그림으로: 어떤 변수가 어디 가나

```cpp
#include "GameFramework/Actor.h"

int g_Counter = 0;          // ← Data (초기화됨)
int g_Buffer[1000];          // ← BSS (초기화 안 됨)
const char* g_Name = "OS";  // ← g_Name 자체는 Data, "OS" 문자열은 Text (읽기 전용)

class AMyActor : public AActor
{
    void Tick(float Dt) override        // ← 함수 코드는 Text
    {
        int Local = 10;                  // ← Stack
        static int s_Calls = 0;          // ← Data (또는 BSS) — Stack 아님 주의!
        s_Calls++;

        TArray<int>* Arr = new TArray<int>();  // ← Arr 변수 자체는 Stack, 가리키는 객체는 Heap
        Arr->SetNum(100);                       //    내부 버퍼도 Heap

        delete Arr;   // Heap 해제. 안 하면 누수.
    }
};
```

### 포인터 변수 vs 가리키는 대상

이게 헷갈리는 사람 많다.

```cpp
void Foo()
{
    int* P = new int(42);
    //  ↑       ↑
    //  Stack   Heap
}
```

- `P` 자체(포인터 변수)는 **Stack**에 있다 (지역 변수니까).
- `P`가 가리키는 `int` 값 42는 **Heap**에 있다.

함수가 끝나면 `P`는 사라지지만, **Heap의 42는 그대로 남는다.** 이게 메모리 누수.

## 4. 같은 exe를 2번 띄우면?

```
프로세스 A (MyGame.exe)              프로세스 B (MyGame.exe)
┌──────────────┐                     ┌──────────────┐
│ Stack: A것    │                     │ Stack: B것    │
│ Heap:  A것    │                     │ Heap:  B것    │
│ BSS/Data: A것 │                     │ BSS/Data: B것 │
│ Text: 공유 가능 ←─── 같은 코드라 ──→ Text: 공유 가능   │
└──────────────┘                     └──────────────┘
```

**Stack, Heap, Data, BSS는 각자 독립.** A의 전역변수 변경이 B에 안 보인다. 다른 주소 공간이니까.

> 다음 강의에서 다룰 **스레드**는 이거랑 다르다. 같은 프로세스 안의 스레드들은 **Heap/Data/BSS를 공유**한다. 스택만 따로. → 그래서 동기화 문제가 생긴다.

## 5. 언리얼에서 자주 만나는 메모리 관련 상황

### (1) UObject는 Heap

```cpp
AMyActor* Actor = NewObject<AMyActor>(this);   // Heap
// 가비지 컬렉터가 수명 관리
```

### (2) USTRUCT는 보통 Stack (값 타입)

```cpp
FVector Pos(1, 2, 3);    // Stack
FRotator Rot(0, 0, 0);   // Stack
```

### (3) TArray의 내부 버퍼는 Heap

```cpp
void Foo()
{
    TArray<int32> Arr;       // Arr 자체(메타데이터)는 Stack
    Arr.Add(1);              // 내부 버퍼는 Heap에 잡힘
}   // 함수 끝 → 소멸자가 Heap 버퍼 해제
```

### (4) 큰 지역 배열은 위험

```cpp
void Render()
{
    FVertex Vertices[100000];   // 위험! 스택 오버플로 가능
    // → TArray나 new로 Heap에 잡아야 함
}
```

## 6. 흔한 오해

### 오해 1: "전역 변수는 무조건 Data 영역"

초기화됐냐 안 됐냐에 따라 Data / BSS로 갈린다. BSS는 exe 파일 용량을 거의 안 먹는다.

### 오해 2: "Stack이 Heap보다 무조건 빠르다"

Stack 할당/해제 자체는 거의 공짜인 게 맞다. 하지만 **이미 할당된 Heap 메모리를 읽고 쓰는 속도**는 Stack과 같다. 빠른 건 "할당/해제"이지 "접근"이 아니다.

### 오해 3: "Heap에 할당하면 OS가 알아서 정리해줌"

**프로세스가 종료되면** OS가 다 회수한다. 하지만 프로세스가 살아있는 동안은 안 풀어준다. 그래서 메모리 누수가 생긴다.

언리얼의 UObject는 **GC(가비지 컬렉터)** 가 정리해주지만, 일반 C++ `new`는 직접 `delete` 해야 한다.

### 오해 4: "스택 오버플로 = 힙 부족"

전혀 다른 에러다.
- **스택 오버플로**: 함수 호출 너무 깊거나 지역 변수 너무 큼. (보통 1~8MB 한도)
- **힙 부족(out of memory)**: `new`로 너무 많이 할당. 시스템 메모리 한도까지 갈 수 있음.

## 7. 면접에서 나오면

**Q1. 프로세스의 메모리 구조를 설명해보세요.**
→ "크게 5개 영역으로 나뉩니다. Text(코드, 읽기 전용), Data(초기화된 전역변수), BSS(초기화 안 된 전역변수, 0으로 채워짐), Heap(동적 할당, 아래에서 위로 자람), Stack(함수 호출/지역변수, 위에서 아래로 자람)입니다."

**Q2. Stack과 Heap의 차이는?**
→ "할당 방식과 수명 관리가 다릅니다. Stack은 함수 호출 시 자동으로 잡히고 리턴 시 자동 해제, 비용이 거의 0입니다. 다만 크기 제한이 있어 보통 1~8MB입니다. Heap은 `new`/`malloc`으로 명시적으로 잡고 명시적으로 해제해야 하며, 할당/해제 비용이 있고 크기는 시스템 메모리만큼 큽니다."

**Q3. 같은 프로그램을 두 번 실행하면 메모리는 어떻게 되나요?**
→ "프로세스가 두 개 생기고 각자 독립된 가상 주소 공간을 가집니다. Stack/Heap/Data/BSS는 별개입니다. 다만 Text(코드) 영역은 읽기 전용이라 OS가 물리 메모리를 공유시킬 수 있습니다."

**Q4. `int* p = new int(5);` 에서 `p`와 `5`는 각각 어디 있나요?**
→ "지역 변수 `p`는 Stack에 있고, `new`로 할당한 `int` 값 5는 Heap에 있습니다. 함수가 끝나면 `p`는 사라지지만 Heap의 5는 남아서 메모리 누수가 됩니다. `delete`로 해제해야 합니다."

**Q5. 스택 오버플로가 발생하는 경우는?**
→ "재귀 호출이 너무 깊거나, 함수 안에 너무 큰 지역 배열을 선언했을 때입니다. 스택은 보통 1~8MB로 제한되어 있고, 함수 호출 시 스택 프레임이 계속 쌓이면 한도를 넘어 크래시가 납니다."

- [ ] 지역 변수, 전역 변수(초기화 O/X), `new`로 잡은 객체가 각각 어느 영역인지 안다.
- [ ] 포인터 변수와 포인터가 가리키는 대상이 다른 영역에 있을 수 있다는 걸 안다.

- [ ] 큰 배열은 왜 지역 변수로 만들면 안 되는지 설명할 수 있다.
