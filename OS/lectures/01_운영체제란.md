# 1강 — 운영체제란 무엇인가?

> **목표:** "OS가 도대체 왜 필요한가?"에 대한 감을 잡는다.
> **선수 지식:** 없음. C++로 `main()` 함수 하나 짜본 적 있으면 충분.

---

## 1. 일단, 컴퓨터에 OS가 없다고 상상해보자

언리얼 에디터를 켜고 게임을 만든다. `Play` 버튼을 누른다. 캐릭터가 화면에 뜬다.

이 단순한 동작 안에서 실제로 일어나는 일을 나열해보면:

- CPU 코어 16개 중 어떤 코어에서 내 게임을 돌릴지 정해야 한다.
- RAM 32GB 중 어디 어디를 내 게임이 써도 되는지 정해야 한다.
- 키보드 입력이 들어오면 누구한테 줄지 정해야 한다.
- GPU에 그림을 그리라고 시켜야 한다.
- 저장 버튼을 누르면 SSD에 바이트를 써야 한다.
- 동시에 디스코드, 크롬, 비주얼 스튜디오도 돌고 있다.

**만약 OS가 없다면?** 이걸 다 게임 코드에서 직접 해야 한다.

```cpp
// OS 없는 세상의 main()
int main()
{
    // 1. 인텔 i7-13700K의 CPUID 명령으로 코어 개수 확인
    // 2. APIC 레지스터에 직접 접근해서 코어 0번에 내 코드 할당
    // 3. RAM 컨트롤러를 두드려서 0x40000000 주소부터 1GB 잡기
    // 4. 키보드 컨트롤러 포트(0x60)를 polling 하는 인터럽트 핸들러 설치
    // 5. NVIDIA GPU의 PCIe 레지스터에 명령어 큐 만들기
    // 6. NVMe SSD의 NVMe 명령 큐를 직접 조작
    // ... 게임 코드는 어디?
}
```

이걸 게임 만들 때마다 한다고? **그것도 마우스 모델, GPU 모델, SSD 모델마다 다르게?**

이걸 대신 해주는 게 운영체제다.

---

## 2. OS의 두 가지 역할

운영체제(Operating System)는 크게 두 가지 일을 한다.

### 역할 1: 자원 관리자 (Resource Manager)

CPU, 메모리, 디스크, 네트워크, 입출력 장치 — 이걸 **여러 프로그램이 사이좋게 나눠 쓰도록** 분배한다.

| 자원 | OS가 하는 일 |
|------|------------|
| CPU | "지금은 게임 차례, 다음은 디스코드 차례" (스케줄링) |
| 메모리 | "게임아 너는 이 32GB 중 8GB만 써" (메모리 관리) |
| 디스크 | "동시에 SSD 읽으려는 놈이 셋이네, 줄 세우자" (I/O 스케줄링) |
| 네트워크 | "이 패킷은 크롬, 이건 게임 클라이언트" (소켓 관리) |

CPU 코어가 16개고 프로세스가 200개면, OS가 **빠르게 돌아가면서** "지금은 너, 지금은 너" 하고 돌려준다. 이걸 우리는 마치 200개가 동시에 도는 것처럼 느낀다.

### 역할 2: 하드웨어 추상화 (Abstraction)

게임 코드가 `로그 텍스트를 화면에 출력하고 싶다` 라고 할 때:

```cpp
UE_LOG(LogTemp, Warning, TEXT("Hello"));
```

이 한 줄을 위해 게임 개발자가 다음을 알 필요가 **없다.**

- 모니터가 4K냐 1080p냐
- 콘솔 창이 cmd냐 PowerShell이냐 Windows Terminal이냐
- 어떤 폰트 파일을 써야 하나
- GPU 드라이버 버전이 뭐냐

OS가 "텍스트 한 줄 출력해줘" 라는 **공통 인터페이스**를 제공하기 때문이다. 우리는 그걸 호출만 하면 된다. 하드웨어가 뭐든, OS가 알아서 번역해준다.

> **핵심 비유:** OS는 게임 개발자와 하드웨어 사이의 **통역사이자 호텔 매니저**다.
> - 통역사: 게임이 "파일 열어줘" 라고 하면 SSD 명령어로 번역
> - 매니저: 다른 손님(프로세스)들과 자원이 겹치지 않게 방을 배정

## 3. 그림으로 보면

```
┌──────────────────────────────────────────────────┐
│  내 게임 (.exe)   디스코드   크롬   비주얼스튜디오    │   ← 응용 프로그램
├──────────────────────────────────────────────────┤
│        Win32 API / DirectX / WinSock 등          │   ← 시스템 콜 (OS의 창구)
├──────────────────────────────────────────────────┤
│              Windows 커널 (OS의 본체)             │   ← 운영체제
│   - 프로세스 스케줄러                              │
│   - 메모리 관리자                                  │
│   - 파일 시스템                                   │
│   - 디바이스 드라이버                              │
├──────────────────────────────────────────────────┤
│   CPU (i7)  │  RAM 32GB  │  RTX 4070  │  SSD     │   ← 하드웨어
└──────────────────────────────────────────────────┘
```

게임은 하드웨어를 **직접** 만지지 않는다. 항상 OS를 거친다.
이게 왜 중요한지는 다음 강의(커널 모드 / 유저 모드)에서 자세히 다룬다.

## 4. 코드로 체감해보기

언리얼에서 우리가 자주 쓰는 함수들이 사실 OS 기능을 부르고 있다.

```cpp
// (1) 메모리 할당 — OS가 RAM에서 영역 잡아줌
TArray<int32> Numbers;
Numbers.Reserve(1000);          // 내부적으로 malloc → OS의 가상 메모리 시스템 호출

// (2) 파일 입출력 — OS가 SSD에서 바이트 읽어줌
FString Text;
FFileHelper::LoadFileToString(Text, TEXT("Config.ini"));   // 내부적으로 ReadFile (Win32 API)

// (3) 시간 — OS가 하드웨어 타이머에서 값 가져옴
double Now = FPlatformTime::Seconds();   // 내부적으로 QueryPerformanceCounter (Win32 API)

// (4) 스레드 생성 — OS가 새 실행 흐름 만들어줌
FRunnableThread* Thread = FRunnableThread::Create(Runnable, TEXT("MyThread"));
//                                                              ↑ 내부적으로 CreateThread (Win32 API)

// (5) 콘솔 출력 — OS가 콘솔 창에 글자 그려줌
UE_LOG(LogTemp, Log, TEXT("Hello"));    // 내부적으로 WriteConsole / WriteFile
```

**언리얼 엔진이 두꺼운 이유 중 하나가 이거다.** 게임 개발자가 `FPlatformTime::Seconds()` 한 줄 부르면 끝이지만, 그 아래에서 Windows/Mac/Linux/PS5/Xbox 별로 **각자 OS API를 다르게 호출**해줘야 한다. 그 다 처리해주는 게 엔진의 플랫폼 추상화 레이어다.

## 5. 흔한 오해

### 오해 1: "OS = Windows 화면 (바탕화면, 작업표시줄)"

바탕화면, 작업표시줄, 시작 메뉴는 **OS의 일부일 뿐이고, 본체가 아니다.** 본체는 보이지 않는 곳에서 도는 **커널(Kernel)**.

리눅스 서버는 GUI 없이도 OS로 잘 돈다. 게임 콘솔(PS5, Xbox)도 OS가 있지만 우리가 보는 화면은 일부일 뿐.

### 오해 2: "OS 없으면 그냥 빠르겠지?"

OS가 자원 관리 때문에 약간의 오버헤드를 만드는 건 맞다. 하지만 OS가 없으면 **하드웨어를 직접 다 다뤄야** 하고, 다른 프로그램과 충돌 처리도 직접 해야 한다. 게임 하나 만들기 위해 운영체제부터 다시 만드는 셈.

> 예외: 임베디드 / 마이크로컨트롤러는 OS 없이 직접 도는 코드가 흔하다. 하지만 거기선 도는 프로그램이 하나뿐이라 자원 분배 문제가 없음.

### 오해 3: "게임 개발자는 OS 몰라도 됨"

언리얼이 많이 가려주는 건 맞다. 하지만:

- **멀티스레드 버그 (Race Condition, Deadlock)** — OS 동기화 개념 모르면 못 잡음
- **메모리 누수, 페이지 폴트로 인한 끊김** — 가상 메모리 모르면 진단 불가
- **비동기 로딩이 왜 필요한가** — 블로킹 I/O 모르면 설계 못 함
- **캐시 친화적 코드** — 캐시 계층 모르면 최적화 못 함

이게 다 이번 강의 시리즈에서 다룰 내용이다.

## 6. 면접에서 나오면

**Q1. 운영체제가 하는 일을 설명해보세요.**
→ "크게 두 가지입니다. 첫째는 자원 관리 — CPU, 메모리, 디스크, I/O 장치를 여러 프로세스가 나눠 쓰게 분배합니다. 둘째는 하드웨어 추상화 — 응용 프로그램이 하드웨어 세부사항을 모르고도 동작할 수 있도록 시스템 콜이라는 공통 인터페이스를 제공합니다."

**Q2. 운영체제가 없으면 어떤 문제가 생길까요?**
→ "각 프로그램이 직접 하드웨어를 다뤄야 하므로, 첫째로 하드웨어 모델마다 코드를 다시 짜야 합니다. 둘째로 여러 프로그램을 동시에 돌릴 수가 없습니다 — 누가 CPU를 쓰고 누가 메모리를 쓰는지 조율해줄 주체가 없으니까요."

**Q3. 커널(Kernel)이 뭔가요?**
→ "OS의 핵심 부분으로, 자원 관리와 하드웨어 제어를 담당하는 코드입니다. 우리가 평소 보는 바탕화면이나 작업 표시줄은 커널 위에서 도는 별도 프로그램이고, 진짜 OS는 그 아래에서 보이지 않게 돕니다."

면접관이 그 다음에 던지기 좋은 질문: *"그럼 응용 프로그램은 커널 코드를 어떻게 호출하나요?"* → 다음 강의(시스템 콜)에서 다룬다.
