# 5강 — 언리얼의 스레드 구조

> **목표:** 언리얼 엔진이 어떤 스레드들로 굴러가는지, 게임 로직과 렌더링이 어떻게 분리되는지 이해한다.
> **선수 지식:** 4강 (프로세스 vs 스레드).

> **이 강의는 OS 개념을 언리얼이라는 실제 케이스에 적용해보는 응용편이다.** 면접에서 "언리얼 멀티스레딩에 대해 아세요?" 들으면 답할 수 있는 게 목표.

---

## 1. 도입 — 한 게임에 스레드가 몇 개나 도는가

언리얼로 만든 게임을 Visual Studio 디버거에 붙여 보면 스레드 창에 30~50개가 떠 있다. 우리가 직접 만든 적도 없는데 왜 이렇게 많을까?

언리얼이 내부적으로 굴리는 주요 스레드들이 다 떠 있어서 그렇다. 큰 그림으로 보면 4개 카테고리:

1. **Game Thread** — 게임 로직 (1개)
2. **Render Thread** — 렌더링 명령 만들기 (1개)
3. **RHI Thread** — GPU에 명령 보내기 (1개)
4. **Worker Thread Pool** — 잡일 (보통 코어 수만큼)

추가로 Audio Thread, Loading Thread, IO Thread 등등이 있다.

---

## 2. Game Thread — "게임의 메인"

우리가 짠 코드 대부분이 도는 곳.

```cpp
void AMyActor::Tick(float DeltaTime)        // ← Game Thread
{
    SetActorLocation(...);                   // ← Game Thread
    PlayAnimation();                         // ← Game Thread
}

void AMyActor::BeginPlay()                   // ← Game Thread
{
    UE_LOG(LogTemp, Log, TEXT("Begin"));
}
```

특징:
- **단 1개.** 언리얼 게임에 Game Thread는 하나뿐.
- 액터, 컴포넌트, UObject 거의 다 여기서만 만지는 게 원칙.
- Blueprint 실행도 여기서.
- 60FPS 게임이면 **한 프레임 = 16.6ms 안에 끝나야** 다음 프레임이 안 밀린다.

> 그래서 Game Thread에서 무거운 작업(파일 로딩, 복잡한 알고리즘) 하면 즉시 끊김(hitch) 발생. → Worker Thread로 빼야 한다.

---

## 3. Render Thread — "렌더링 명령을 만들어라"

Game Thread는 "캐릭터 위치는 (1,2,3), 이 메시 그려" 같은 **명령**을 만들고, 실제 렌더링은 Render Thread가 한다.

왜 분리? Game Thread가 다음 프레임의 로직을 돌리는 **동안** Render Thread는 현재 프레임을 그릴 수 있다. **파이프라이닝**으로 처리량을 올리는 것.

```
시간 →
Game Thread:    [Frame N 계산][Frame N+1 계산][Frame N+2 계산]
Render Thread:                [Frame N 렌더] [Frame N+1 렌더] [Frame N+2 렌더]
```

즉 **Render Thread는 항상 Game Thread보다 1프레임 늦게 본다.** 그래서 Game Thread에서 만든 데이터를 Render Thread에 안전하게 넘기는 패턴이 있다.

```cpp
// Game Thread에서:
ENQUEUE_RENDER_COMMAND(MyCommand)(
    [Data = LocalData](FRHICommandListImmediate& RHICmdList)
    {
        // 이 람다는 Render Thread에서 나중에 실행됨
        // Data는 복사돼서 안전
    }
);
```

- `ENQUEUE_RENDER_COMMAND` = "이 작업을 Render Thread의 큐에 넣어줘."
- 람다 캡처는 **복사**가 원칙. 포인터 그대로 넘기면 Game Thread가 그 동안 바꿔버려서 Race Condition.

---

## 4. RHI Thread — "GPU랑 진짜 대화"

RHI(Rendering Hardware Interface)는 DirectX/Vulkan/Metal 같은 그래픽스 API를 추상화한 층이다.

- Render Thread는 "메시 그려, 머터리얼 바꿔" 같은 **추상 명령**을 만든다.
- RHI Thread는 그걸 받아서 **실제 DirectX 11/12, Vulkan 호출**로 번역하고 GPU에 보낸다.

또 한 번 분리한 이유: DX/Vulkan API 호출 자체가 무거울 수 있어서, Render Thread가 다음 명령 만드는 동안 RHI Thread가 GPU에 명령 송신.

```
Game Thread:    [로직]
                  ↓ (커맨드 큐)
Render Thread:  [렌더 명령 만들기]
                  ↓ (RHI 명령 큐)
RHI Thread:     [DX12 호출 → GPU]
                  ↓
GPU:            [실제 픽셀 그리기]
```

총 **3프레임 정도** 파이프라인이 형성된다. 그래서 마우스 클릭 → 화면에 반영까지 약간의 입력 지연이 생기는 거다.

> 보너스: 모바일이나 저사양에서는 RHI Thread를 안 만들고 Render Thread가 직접 그리는 모드도 있다.

## 5. Worker Thread Pool — "잡일 처리반"

언리얼은 **TaskGraph** 와 **FRunnable** 두 가지 방식으로 워커 스레드를 굴린다.

### TaskGraph (대부분 이걸 씀)

병렬화 가능한 작은 작업을 큐에 넣으면 **워커 스레드들이 알아서 가져가서 처리**한다.

```cpp
AsyncTask(ENamedThreads::AnyBackgroundThreadNormalTask, []()
{
    // 워커 스레드에서 실행됨
    HeavyComputation();

    // 결과를 다시 Game Thread로 넘기고 싶으면:
    AsyncTask(ENamedThreads::GameThread, []()
    {
        UpdateUI();   // 이건 Game Thread에서 실행돼야 안전
    });
});
```

쓸 만한 케이스:
- 경로 탐색(A*) 백그라운드로 돌리기
- 절차적 생성 (지형, 던전)
- JSON/Save 파일 파싱

### FRunnable — 평생 도는 워커가 필요할 때

```cpp
class FMyWorker : public FRunnable
{
public:
    virtual uint32 Run() override
    {
        while (!bStop)
        {
            DoWork();
            FPlatformProcess::Sleep(0.016f);
        }
        return 0;
    }

    virtual void Stop() override { bStop = true; }

private:
    FThreadSafeBool bStop = false;
};

// 사용:
FMyWorker* W = new FMyWorker();
FRunnableThread* T = FRunnableThread::Create(W, TEXT("MyWorker"));
```

쓸 만한 케이스:
- 네트워크 수신 루프
- 백그라운드 파일 와처(watcher)
- 음성 채팅 인코딩

```

                  │       프로세스 (MyGame.exe)        │

                  │  │  - 렌더 명령 빌드            │  │

                  │  └──────┬──────────────────────┘  │
                  │         │                         │
                  │         ↓                         │
                  │  ┌─────────────────────────────┐  │

                  │  │  - DX12/Vulkan 호출          │  │
                  │  └──────┬──────────────────────┘  │
                  │         │                         │
                  │  ┌─────────────────────────────┐  │
                  │  │ Worker Pool (N개)           │  │
                  │  │  - TaskGraph                │  │
                  │  │  - 비동기 로딩, AI 등        │  │

                  │                                   │
```

## 7. 실전 — Game Thread에서 하면 안 되는 일

```cpp
// 나쁜 예 1: 큰 파일 동기 로딩
void AMyActor::BeginPlay()
{
    FString Huge;
    FFileHelper::LoadFileToString(Huge, TEXT("HugeFile.txt"));  // 100ms 걸리면 게임 6프레임 멈춤
}

// 좋은 예 1: 비동기로 빼기
void AMyActor::BeginPlay()
{
    AsyncTask(ENamedThreads::AnyBackgroundThreadNormalTask, [this]()
    {
        FString Huge;
        FFileHelper::LoadFileToString(Huge, TEXT("HugeFile.txt"));

        AsyncTask(ENamedThreads::GameThread, [this, Huge]()
        {
            // 결과 사용은 다시 Game Thread에서
            ApplyContent(Huge);
        });
    });
}
```

```cpp
// 나쁜 예 2: 워커 스레드에서 UObject 만들기
AsyncTask(ENamedThreads::AnyBackgroundThreadNormalTask, []()
{
    AMyActor* Actor = NewObject<AMyActor>();  // ❌ 크래시 가능!
});
// UObject 생성/소멸은 Game Thread 전용. GC와 충돌.

// 나쁜 예 3: 워커 스레드에서 UI 업데이트
AsyncTask(ENamedThreads::AnyBackgroundThreadNormalTask, []()
{
    MyWidget->SetText(TEXT("Done"));  // ❌ Slate는 Game Thread 전용
});
```

**원칙:** UObject, Actor, Component, Slate UI — **거의 다 Game Thread 전용.**
워커는 **순수 데이터 계산** 만.

## 8. OS 개념과 연결지어보기

| 언리얼 개념 | OS 개념 |
|------------|---------|
| Game/Render/RHI Thread 분리 | 멀티스레드 (4강) |
| Worker Thread Pool | 스레드 풀 패턴, 컨텍스트 스위칭 비용 줄이기 (6강) |
| `FCriticalSection`, `FScopeLock` | Mutex (8강) |
| `FThreadSafeBool`, `TAtomic` | Atomic (9강) |
| `ENQUEUE_RENDER_COMMAND` | 메시지 큐 패턴 (스레드 간 안전한 데이터 전달) |
| `FStreamableManager` 비동기 로딩 | 비동기 I/O (15강) |

다음 강의들에서 이 개념들을 하나씩 깊게 본다. 이번 강의는 "언리얼에서 OS 개념이 이렇게 쓰이는구나"를 깔아두는 강의.

## 9. 흔한 오해

### 오해 1: "Tick은 매 프레임 정확히 16.6ms마다 호출된다"

아니다. Tick은 **이전 프레임이 끝났을 때** 호출된다. 무거우면 그만큼 늦게 다음 Tick. `DeltaTime`이 그래서 가변.

### 오해 2: "Render Thread도 내가 코드 짤 수 있다"

직접은 거의 안 짠다. `ENQUEUE_RENDER_COMMAND` 안의 람다가 Render Thread에서 도는 정도. 머터리얼이나 셰이더 작업이 아니면 게임 개발자는 신경 안 써도 됨.

### 오해 3: "워커 스레드 많이 만들수록 좋다"

NO. 코어 수보다 많이 만들면 컨텍스트 스위칭 오버헤드만 생긴다. 언리얼이 자동으로 코어 수에 맞춰 워커 풀 크기를 정한다.

### 오해 4: "멀티스레드면 동시에 같은 메모리 만져도 됨"

이 오해 때문에 게임이 죽는다. 7~10강에서 다룸. **공유 데이터는 반드시 동기화** 가 필요.

## 10. 면접에서 나오면

**Q1. 언리얼은 어떤 스레드 구조로 동작하나요?**
→ "크게 Game Thread, Render Thread, RHI Thread, Worker Thread Pool이 있습니다. Game Thread는 게임 로직, Render Thread는 렌더 명령 생성, RHI Thread는 DirectX/Vulkan 호출 송신, Worker Pool은 TaskGraph로 들어온 백그라운드 작업을 처리합니다."

**Q2. Game Thread와 Render Thread는 왜 분리되어 있나요?**
→ "파이프라이닝으로 처리량을 올리기 위해서입니다. Render Thread가 N 프레임을 그리는 동안 Game Thread는 N+1 프레임의 로직을 미리 계산합니다. 그래서 Render Thread는 Game Thread보다 1프레임 정도 뒤를 봅니다."

**Q3. Worker Thread에서 UObject를 만들면 어떻게 되나요?**
→ "위험합니다. UObject 생성/소멸은 Game Thread와 가비지 컬렉터가 관리하기 때문에, 워커 스레드에서 NewObject를 하면 GC와 충돌해 크래시가 날 수 있습니다. 워커에서는 순수 데이터 계산만 하고, 결과를 Game Thread로 다시 던져서 UObject를 만드는 게 원칙입니다."

**Q4. 무거운 파일 로딩을 Game Thread에서 직접 하면 어떤 문제가 생기나요?**
→ "Game Thread가 블로킹되어 그 시간 동안 Tick이 멈춥니다. 100ms 걸리면 60FPS 기준 6프레임이 통째로 끊겨 hitching이 발생합니다. 비동기 워커로 빼고 결과만 Game Thread에서 받아야 합니다."

**Q5. `ENQUEUE_RENDER_COMMAND`가 왜 필요한가요?**
→ "Game Thread와 Render Thread는 다른 스레드라서 직접 접근하면 Race Condition이 납니다. ENQUEUE_RENDER_COMMAND는 Game Thread에서 만든 데이터를 큐에 넣어두고 Render Thread가 안전하게 꺼내 처리하게 하는 메시지 패턴입니다."

## 11. 셀프 체크

- [ ] Game/Render/RHI Thread의 역할을 각각 한 문장씩 말할 수 있다.
- [ ] Render Thread가 Game Thread보다 늦게 본다는 게 무슨 뜻인지 안다.
- [ ] Game Thread에서 하면 안 되는 일을 두 가지 이상 말할 수 있다.
- [ ] 워커 스레드에서 UObject를 만들면 안 되는 이유를 안다.
- [ ] `ENQUEUE_RENDER_COMMAND` 가 왜 필요한지 설명할 수 있다.
