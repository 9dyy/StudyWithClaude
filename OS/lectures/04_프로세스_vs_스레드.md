# 4강 — 프로세스 vs 스레드

> **목표:** 프로세스와 스레드의 진짜 차이를 설명할 수 있게 된다. 왜 둘 다 필요한가?
> **선수 지식:** 3강 (프로세스 메모리 구조).

---

## 1. 도입 — 게임은 왜 멀티스레드인가?

언리얼로 만든 게임에서 일어나는 일을 떠올려보자.

- 캐릭터 위치 업데이트
- 화면 렌더링
- 텍스처/모델 비동기 로딩
- 서버와 패킷 주고받기
- 사운드 믹싱
- 물리 시뮬레이션
- 애니메이션 블렌딩

이걸 **한 줄로 순서대로** 다 처리한다고 해보자.

```cpp
while (true)  // 메인 루프
{
    UpdateInput();      // 1ms
    UpdatePhysics();    // 5ms
    UpdateAI();         // 3ms
    UpdateAnimation();  // 4ms
    Render();           // 10ms   ← 여기서 GPU 기다리는 동안 CPU 놀음
    SendNetwork();      // 2ms
    LoadAssets();       // 100ms  ← 여기서 디스크 기다리는 동안 다 멈춤
    // 합: 125ms → 8 FPS. 게임 끝.
}
```

문제:
1. 한 작업이 느리면 모든 게 멈춘다 (에셋 로딩 중에는 캐릭터도 못 움직임).
2. CPU 코어가 16개여도 1개밖에 못 쓴다.

해결: **여러 일을 동시에.** 그게 멀티스레드.

---

## 2. 프로세스 (Process)

3강에서 본 그거. **실행 중인 프로그램 하나.**

```
프로세스 A (MyGame.exe)
├─ 자기만의 가상 주소 공간 (Text/Data/BSS/Heap/Stack)
├─ 열어둔 파일 핸들
├─ 환경 변수
├─ 현재 디렉토리
└─ 보안 컨텍스트 (어느 유저가 실행했나)
```

특징:
- **메모리 공간이 독립적이다.** A의 변수를 B가 못 본다.
- 그래서 **안전하다.** 한 프로세스가 죽어도 다른 프로세스는 멀쩡.
- 그래서 **통신이 어렵다.** IPC(Inter-Process Communication: 파이프, 공유 메모리, 소켓 등)가 필요.
- 만들고 없애는 비용이 **비싸다.** (메모리 공간 통째로 잡아야 함)

---

## 3. 스레드 (Thread)

**같은 프로세스 안에서 도는 실행 흐름.** 한 프로세스에 여러 개 있을 수 있다.

```
프로세스 (MyGame.exe)
├─ Text/Data/BSS/Heap   ← 스레드들이 공유
├─ 파일 핸들 등         ← 스레드들이 공유
│
├─ 스레드 1 (Game Thread)
│   └─ 자기만의 Stack, 레지스터, PC(Program Counter)
├─ 스레드 2 (Render Thread)
│   └─ 자기만의 Stack, 레지스터, PC
├─ 스레드 3 (Audio Thread)
│   └─ 자기만의 Stack, 레지스터, PC
└─ 스레드 4 (Worker)
    └─ 자기만의 Stack, 레지스터, PC
```

특징:
- **Heap, Data, BSS, 코드를 공유.** 전역 변수 하나 바꾸면 다른 스레드도 영향 받음.
- **Stack과 레지스터(실행 상태)만 따로.**
- 만들고 없애는 비용이 **싸다.** (프로세스 만들기보단 훨씬)
- 통신이 **쉽다.** (그냥 전역 변수, 힙 객체로 데이터 주고받음)
- 그래서 **위험하다.** → 동기화 문제 (다음 강의들 핵심)

---

## 4. 그림으로 비교

```
[ 단일 프로세스, 단일 스레드 ]
프로세스
└─ 스레드 1 ── PC ─→ 코드 한 줄씩 실행

[ 단일 프로세스, 멀티 스레드 ]  ← 언리얼 게임
프로세스 (공유 메모리)
├─ 스레드 1 ── PC ─→ 게임 로직 실행 중
├─ 스레드 2 ── PC ─→ 렌더 명령 만드는 중
└─ 스레드 3 ── PC ─→ 에셋 디코딩 중

[ 멀티 프로세스 ]  ← 크롬 브라우저 같은 거
프로세스 A (탭 1)    프로세스 B (탭 2)    프로세스 C (확장 프로그램)
└─ 메모리 독립        └─ 메모리 독립        └─ 메모리 독립
```

## 5. 코드로 — 같은 프로세스의 두 스레드는 메모리를 진짜로 공유한다

```cpp
#include "HAL/Runnable.h"
#include "HAL/RunnableThread.h"

int32 g_Shared = 0;   // 전역 변수 — 모든 스레드가 공유 (BSS/Data)

class FWorker : public FRunnable
{
public:
    virtual uint32 Run() override
    {
        for (int i = 0; i < 1000000; ++i)
        {
            g_Shared++;   // 같은 메모리 주소를 여러 스레드가 같이 건드림
        }
        return 0;
    }
};

void Test()
{
    FWorker* W1 = new FWorker();
    FWorker* W2 = new FWorker();

    FRunnableThread* T1 = FRunnableThread::Create(W1, TEXT("W1"));
    FRunnableThread* T2 = FRunnableThread::Create(W2, TEXT("W2"));

    T1->WaitForCompletion();
    T2->WaitForCompletion();

    UE_LOG(LogTemp, Warning, TEXT("Result = %d"), g_Shared);
    // 기대값: 2,000,000
    // 실제값: 매번 다름. 1,200,000 정도 나올 수 있음. ← Race Condition!
}
```

이게 7강에서 자세히 다룰 **Race Condition** 의 정체다. `g_Shared++` 는 "읽고, 1 더하고, 쓰고" 인데 두 스레드가 동시에 하면 서로 덮어쓴다.

> 만약 이걸 **별개 프로세스 두 개**로 했다면? 각자 자기 `g_Shared`를 가지니까 충돌이 없다. 둘 다 1,000,000 으로 끝난다. 그런데 둘이 합산해서 결과를 보려면 별도 통신(IPC)이 필요하다.

## 6. 비교 표

| 항목 | 프로세스 | 스레드 |
|------|---------|--------|
| 메모리 공간 | 독립 | Heap/Data 공유, Stack만 독립 |
| 생성 비용 | 비쌈 (수십 ms) | 쌈 (수 ms 이하) |
| 컨텍스트 스위칭 비용 | 비쌈 (TLB 무효화 등) | 상대적으로 쌈 |
| 통신 | IPC 필요 (파이프, 소켓, 공유 메모리...) | 그냥 메모리 공유 |
| 안전성 | 하나 죽어도 다른 건 OK | 하나 죽으면 프로세스 전체 죽음 |
| 동기화 문제 | 거의 없음 | 매우 중요 (Mutex, Atomic 등) |

## 7. 언제 프로세스, 언제 스레드?

### 스레드를 쓰는 경우

- **같은 데이터를 여러 작업이 빠르게 공유해야 할 때**
- 게임처럼 한 화면을 위해 여러 일이 협력해야 할 때
- 대부분의 게임 클라이언트, 게임 서버는 멀티스레드.

### 프로세스를 쓰는 경우

- **격리(Isolation)** 가 중요할 때
  - 크롬: 탭 하나가 죽어도 다른 탭 안 죽게 → 탭마다 프로세스
  - 게임 런처가 게임 본체와 분리되어 있는 것 (런처가 죽어도 게임은 OK)
- **다른 권한이 필요할 때**
- **장애 격리**가 중요할 때

### 게임 개발자의 현실

게임 클라이언트는 **한 프로세스 + 많은 스레드** 구조가 거의 표준이다. 언리얼도 그렇다.
프로세스는 보통 런처 ↔ 게임, 게임 ↔ 안티치트(예: BattlEye) 처럼 **격리**가 필요할 때만.

## 8. PCB와 TCB — OS가 기억하는 것들

OS는 모든 프로세스와 스레드를 추적하기 위해 자료구조를 들고 있다.

- **PCB (Process Control Block)**: 프로세스 하나당 1개
  - PID, 가상 주소 공간 정보, 파일 핸들 목록, 보안 컨텍스트 등
- **TCB (Thread Control Block)**: 스레드 하나당 1개
  - 스레드 ID, **레지스터 상태**, **스택 포인터**, 우선순위 등

CPU가 스레드를 바꿔 돌릴 때(컨텍스트 스위칭, 6강), TCB에서 레지스터 값을 꺼내거나 저장한다.

> 면접 팁: "PCB/TCB가 뭐냐?" 물어보면 "OS가 프로세스/스레드를 관리하기 위한 자료구조이고, 컨텍스트 스위칭 시 여기 저장된 레지스터 상태를 꺼내 씁니다" 정도면 충분.

## 9. 흔한 오해

### 오해 1: "스레드를 많이 만들수록 빠르다"

전혀 아니다. 코어가 16개인데 스레드 100개를 만들면 OS가 미친 듯이 컨텍스트 스위칭만 한다. 오히려 느려진다. (6강에서 자세히)

게임은 보통 **코어 수 + α** 만큼만 스레드 만든다. 언리얼의 Task Graph도 이걸 자동 관리.

### 오해 2: "전역 변수만 공유되고 지역 변수는 안 공유된다"

지역 변수는 **각 스레드의 Stack에 따로 잡힌다.** 한 함수 안에서 만든 지역 변수는 그 스레드만의 것.
다만 그 지역 변수의 **주소를 다른 스레드에 넘기면** 공유된다 (그 주소를 읽을 수 있으니까).

```cpp
void Foo()
{
    int Local = 42;
    AnotherThreadAccess(&Local);  // 위험! Foo가 끝나면 Local은 사라지는데 다른 스레드가 그 주소를 읽으면?
}
```

### 오해 3: "프로세스가 스레드보다 항상 안전하다"

격리 측면에선 맞지만, **두 프로세스가 공유 메모리(IPC의 한 형태)를 쓰면** 똑같이 동기화 문제가 생긴다. 안전한 건 "메모리 공간이 분리되어 있을 때까지"의 얘기.

### 오해 4: "main 함수도 스레드다"

**맞다.** `main()`이 도는 스레드를 **메인 스레드**라고 부른다. 모든 프로세스에는 최소 1개의 스레드가 있다(메인 스레드). 추가 스레드를 만들지 않아도 이미 1개.

## 10. 면접에서 나오면

**Q1. 프로세스와 스레드의 차이를 설명해보세요.**
→ "프로세스는 실행 중인 프로그램으로 각자 독립된 가상 주소 공간을 가집니다. 스레드는 한 프로세스 안에서 도는 실행 흐름으로, 같은 프로세스 안의 스레드끼리는 Heap, Data, 코드 영역을 공유하고 Stack과 레지스터만 따로 가집니다. 그래서 스레드는 통신이 쉽지만 동기화 문제가 생기고, 프로세스는 격리되어 안전하지만 통신에 IPC가 필요합니다."

**Q2. 멀티스레딩의 장단점은?**
→ "장점은 같은 데이터를 공유하면서 동시에 여러 작업을 할 수 있어 CPU를 잘 활용할 수 있고, 한 작업이 블로킹돼도 다른 작업이 계속 도는 점입니다. 단점은 공유 자원에 대한 Race Condition, Deadlock 같은 동기화 문제가 생기고, 디버깅이 어렵고, 한 스레드가 죽으면 프로세스 전체가 죽는다는 점입니다."

**Q3. 프로세스를 새로 만드는 게 스레드보다 비싼 이유는?**
→ "프로세스 생성은 새로운 가상 주소 공간을 만들고, 페이지 테이블을 세팅하고, 부모 프로세스의 자원을 복사하거나 새 exe를 로드해야 합니다. 스레드는 이미 있는 프로세스에 스택과 TCB만 추가하면 되므로 훨씬 가볍습니다."

**Q4. 컨텍스트 스위칭 비용이 프로세스 간 vs 스레드 간 어떻게 다른가요?**
→ "같은 프로세스 안의 스레드끼리 전환할 때는 가상 주소 공간이 같으므로 TLB(주소 변환 캐시)를 유지할 수 있습니다. 다른 프로세스로 전환하면 가상 주소 공간이 달라지므로 TLB를 무효화해야 하고, 캐시 적중률도 떨어집니다. 그래서 프로세스 간 스위칭이 더 비쌉니다."

**Q5. 언리얼 같은 게임 엔진이 멀티프로세스가 아니라 멀티스레드를 쓰는 이유는?**
→ "게임 로직, 렌더링, 물리, 사운드, 에셋 로딩 같은 작업이 같은 게임 상태(예: 액터 위치, 트랜스폼)를 빠르게 공유해야 합니다. 프로세스로 나누면 IPC 비용이 너무 커서 60FPS를 못 맞춥니다. 격리가 필요한 부분(런처, 안티치트)에만 별도 프로세스를 씁니다."

## 11. 셀프 체크

- [ ] 프로세스와 스레드가 메모리를 공유하는지 여부를 말할 수 있다.
- [ ] 스레드끼리 공유되는 영역과 따로 갖는 영역을 구분할 수 있다.
- [ ] 멀티스레드의 장단점을 각각 두 개 이상 댈 수 있다.
- [ ] 같은 프로세스 안 스레드끼리의 컨텍스트 스위칭이 왜 더 싼지 안다.
- [ ] 게임이 멀티프로세스 대신 멀티스레드를 쓰는 이유를 설명할 수 있다.
